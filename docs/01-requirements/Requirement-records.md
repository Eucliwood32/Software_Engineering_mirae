## 1. 문서 개요

### 1.1 목적과 문장 규약
본 문서는 Problem Statement가 정의한 문제와 ConOps가 정의한 운영 시나리오로부터 검증 가능한 요구사항을 도출하고, 그 추적성을 박제한다.

**문장 규약.** 본 문서의 모든 요구사항 문장은 *의무 사양*으로 해석된다. 표기상 한국어 종결형 "~한다"를 사용하되, 이는 모두 "~해야 한다"의 의미다. 수용기준은 Pass/Fail이 명확히 판정 가능한 형태로 글머리표로 분리하여 기술한다.

### 1.2 SRS와의 관계 (폐지)
초기에는 SRS v2.0이 본 문서보다 먼저 작성되어 본 문서가 그에 대한 정합성 감사 역할을 했고, SRS는 본 문서의 결정을 반영해 v2.2로 동기화되어 왔다. 그러나 SRS는 머리말에 "개발 시 무시"로 표기된 보조 문서였고 모든 수용기준을 본 RR에 위임("RR FR-x.x와 동일")해 왔으므로, **SRS.md는 2026-05-31 폐지(삭제)되었다.** SRS에만 존재하던 요구(이상 신호 예외 처리 = 구 SRS FR-4.2c)는 본 문서 FR-4.2c로 상속·복원하였다(§7 v1.5). 이후 **본 RR이 단일 요구사항 정본**이며, 설계·검증은 `docs/02-design-planning/*`·`docs/03-verification-validation/*`이 담당한다.

### 1.4 용어
- **종합 기여 지표(Aggregate Contribution Indicator):** 정규화된 Git·문서·메신저 지표를 가중치로 결합한 0~1 사이의 *참고용 수치*. 평가 결론이 아니며, 조장의 판단 근거다(STR-7, ConOps P5).
- **미매핑 식별자(Unmapped Identifier):** 입력 소스에서 발견되었으나 조장이 어느 팀원에도 연결하지 않은 식별자. FR-1.2의 "Unknown"과 구분된다.
- **Unknown 작성자:** OOXML 메타데이터 자체가 비어 있는 문서의 작성자 분류. 단일 정의된 분류이며 미매핑 식별자와 다르다.
- **이상 신호(Anomaly Signal):** 시스템이 통계·규칙으로 식별하여 조장에게 표시하는 검토 권유. 판정·결론이 아니다.
- **소통량(Communication Volume):** 산점도에서 사용되는 팀원별 메신저 기여 척도. 정규화된 유효 메시지 수(FR-3.3 자동 불용어 처리 후)로 정의한다.
- **문서 작업 밀도(Document Work Density):** 산점도에서 사용되는 팀원별 문서 기여 척도. 정규화된 문서 글자수로 정의한다.
- **각 점수별 정량적 세부 구성요소 (Raw Data):** 최종 점수 산출의 근거가 되는 원시 수치(Git 커밋 수 및 라인 양, 문서 유효 글자 수 및 도형 수, 메신저 유효 문자 및 발화 횟수). 평가의 투명성을 위해 UI에 노출되어야 한다.

## 4. 기능 요구사항

### FR-1. 통합 입력 — 문서

#### FR-1.1 OOXML/HWPX 텍스트·도형 추출
- **무엇을:** 시스템은 `.pptx`/`.docx`/`.hwpx` 파일에서 공백·개행을 제외한 유효 글자수와 도형 개수를 추출한다.
- **출처:** STR-1, L1, O-1
- **수용기준:**
  - 200자 .docx 입력 → 글자수 200±0.
  - 슬라이드 3장 × 박스 2 × 50자 .pptx → 글자수 300, 도형 수 6.
  - 200자 .hwpx 입력 → 글자수 200±0 (HWPX 표준 스키마 기준).
  - .pptx, .docx, .hwpx가 한 폴더에 혼재된 입력 → 세 형식 모두에서 추출 정상 수행.
  - 빈 파일 → 글자수 0, 도형수 0, 예외 없음.
  - 손상된 파일 → 해당 파일 skip + 경고 메시지에 파일명 포함, 분석 계속.
- **시나리오:** SC-A 2단계

#### FR-1.2 작성자별 집계
- **무엇을:** 시스템은 문서 파일의 메타데이터(OOXML: core_properties, HWPX: 표준 메타 스키마)의 작성자 식별자를 기준으로 글자수를 집계한다. 메타데이터가 부재한 파일은 "Unknown" 분류로 묶는다.
- **출처:** STR-1, L1
- **수용기준:**
  - Alice 100자 + Bob 150자 → `{"Alice":100,"Bob":150}`.
  - 메타 없음 → "Unknown" 키 존재, 예외 없음.
  - "Unknown"은 단일 정의된 분류이며, FR-1.3의 미매핑 식별자와는 다른 개념이다.
- **시나리오:** SC-A 2단계

#### FR-1.3 신원 매핑 (Global Alias Mapping)
- **무엇을:** 시스템은 입력 소스에서 발견된 식별자(계정·이메일·이름)를 조장에게 제시하고, 조장이 하나의 팀원에 여러 식별자를 수동 연결할 수 있게 한다(N:1). **매핑 UI의 제시 시점은 분석 *후* 결과 화면이다**(v1.4, FR-5.7). 즉 1차 분석은 매핑 없이 각 식별자를 독립 인물로 산출하고, 조장이 결과 화면에서 동일인의 식별자들을 골라 병합하면 시스템이 재집계한다. N:1 합산 로직(Model `AliasMapper`)은 동일하며 UI 시점만 이동한다.
- **출처:** STR-3, L3, O-1, SC-C
- **수용기준:**
  - Git·메신저·문서에서 추출되어 결과에 표시된 모든 인물(식별자)이 병합 UI에 빠짐없이 제시된다.
  - 한 팀원에 식별자 3개를 연결(병합) 시, 모든 결과 표시에서 그 셋의 기여가 단일 인격으로 합산되며, 합산은 원시 지표 재집계→재정규화를 거쳐 반영된다(FR-5.7).
  - 병합하지 않은(미선택) 식별자는 각자 독립 인물로 유지된다.
  - 미매핑/미병합 식별자 표시에 각 식별자의 활동 규모(커밋 수·메시지 수 등)가 함께 표시된다.
  - 서로 다른 식별자는 시스템에 의해 임의로 병합되지 않는다(조장의 명시 선택만).
  - 시스템은 식별자 정규화(공백·`. _ -` 제거, 이메일 로컬파트 추출, 대소문자 무시) 기반으로 동일인으로 추정되는 식별자 그룹을 **병합 후보로 제안**할 수 있다. 제안은 결정론적이며(NFR-1.3), 실제 병합은 조장의 확정으로만 이루어진다(자동 병합 금지와 모순되지 않음).
  - FR-1.2의 "Unknown" 분류와 미병합 식별자는 별개로 처리된다.
- **시나리오:** SC-A 5단계(결과 화면 병합), SC-C 전체
- **변경(v1.4):** 분석-전 매핑 모달을 폐기하고 결과 화면 병합(FR-5.7)으로 일원화. 사전 매핑 가정이 사라진 대신 결과를 보며 병합하는 흐름으로 단순화.

### FR-2. 통합 입력 — Git

#### FR-2.1 Git 로그 추출
- **무엇을:** 시스템은 로컬 Git 저장소에서 작성자별 커밋 수·추가 라인·삭제 라인을 수집한다. 잘못된 경로 입력 시 예외를 상위로 전파하지 않고 빈 결과를 반환한다. Git 저장소는 현재 실행 환경을 기준으로 자동 탐지되지 않으며, 조장이 명시적으로 선택(또는 드래그앤드롭)한 경로에 한해서만 분석한다.
- **출처:** STR-1, L1, O-1
- **수용기준:**
  - 3커밋(alice@test.com, +10/-5) 저장소 → 정확 일치.
  - 잘못된 경로 → 빈 결과, 예외 없음.
  - 10,000커밋 저장소 → 30초 이내 수집 완료.
- **시나리오:** SC-A 3단계

#### FR-2.2 Git 부재 시 안내
- **무엇을:** 메인 윈도우 표시 전 Git 가용성을 점검하고, 부재 시 모달 안내를 표시한 뒤 Git 기능만 비활성화한 채 앱을 계속 실행한다.
- **출처:** STR-1, ASM-1, C-9
- **수용기준:**
  - Git 미설치 환경 → 메인 윈도우 표시 전 모달 표시.
  - 모달에 "Git이 설치되어 있지 않거나 PATH에 등록되지 않았습니다." 문구를 포함한다.
  - 모달에 다운로드 링크 `https://git-scm.com/download/win`이 포함되고, 클릭 시 OS 기본 브라우저로 열린다.
  - 모달에 PATH 설정 안내 텍스트 1줄 이상이 포함된다.
  - 확인 버튼 클릭 → 앱 정상 실행, Git 기능만 비활성화.
  - Git 정상 설치 환경 → 모달 미표시.
- **시나리오:** SC-A 진입

### FR-3. 통합 입력 — 메신저

#### FR-3.1 카카오톡 파싱
- **무엇을:** 시스템은 카카오톡 `.txt` 내보내기 파일을 발화자·시각·메시지 구조로 정규화한다.
- **출처:** STR-1, L1
- **수용기준:**
  - 카카오톡 표준 형식 정확 파싱(발화자·메시지 분리).
  - UTF-8 인코딩 카카오톡 파일 → 한글 무손상.
  - CP949 인코딩 카카오톡 파일 → 한글 무손상.
  - 카카오톡 단독 데이터 소스만 입력된 상태에서 분석을 실행해도 앱이 멈추지 않고 파이프라인이 정상 작동해야 한다.
  - 카카오톡 로그 파일이 파싱될 때, 추출된 모든 발화자 식별자는 병합(매핑) 목록에 누락 없이 표출되어야 한다.
- **시나리오:** SC-A 4단계

#### FR-3.2 오염 줄 방어적 Skip
- **무엇을:** 시스템은 파싱 실패 줄을 카운트하며 건너뛰고, 정상 레코드와 skip 횟수를 함께 반환한다.
- **출처:** STR-1, STR-6
- **수용기준:**
  - 정상 10 + 오염 3 → records 10, skipped 3.
  - 전체 100줄 오염 → records 0, skipped 100, 예외 없음.
  - 모든 경우에 프로세스가 종료되지 않는다.

#### FR-3.3 자동 불용어 처리 (Automatic Stopword Filtering)
- **무엇을:** 시스템은 의미가 희박한 표현을 *자동으로 식별하여* 유효 메시지 집계에서 제외한다. 사용자(조장)는 불용어 사전을 편집할 수 없다.
- **출처:** STR-4, L4, C-7
- **수용기준:**
  - 시스템은 다음 카테고리의 표현을 자동으로 불용어로 분류하여 집계에서 제외한다.
    - 단순 리액션: `ㅇㅇ`, `ㅋㅋ`, `ㅎㅎ`, `ㄱㄱ`, `ㄴㄴ`, `ㄷㄷ`, `ㅠㅠ` 등 자음·반복 표현.
    - 미디어 태그: `(이모티콘)`, `(사진)`, `(동영상)`, `(파일)`.
    - 1글자 단순 응답: `네`, `예`, `응`, `굳`.
  - 불용어로 분류된 표현은 유효 메시지 수·유효 글자수 집계에서 제외된다.
  - 시스템은 사용자(조장)에게 불용어 사전 편집 UI를 제공하지 않는다.
  - 동일 입력에 대해 동일 불용어 분류 결과가 산출된다 (NFR-1.3 결정론 부합).

### FR-4. 분석 — 정규화·이상 신호·가중치

#### FR-4.1 Min-Max 정규화
- **무엇을:** 시스템은 이질적 단위 지표 3종(Git 라인·문서 글자·메신저 발화)을 0~1 척도로 정규화한다. 분산 0인 경우 전원 0.5. 결과는 소수점 4자리로 반올림한다.
- **출처:** STR-1, STR-8
- **수용기준:**
  - 입력 [0,50,100] → [0.0, 0.5, 1.0].
  - 입력 [75,75,75] → [0.5, 0.5, 0.5], ZeroDivisionError 없음.
  - 모든 결과 0.0 ≤ v ≤ 1.0.

#### FR-4.2 이상치 Capping과 신호 표시
- **무엇을:** 시스템은 단일 커밋의 추가 라인이 1,000줄을 초과할 경우 집계상 1,000줄로 제한한다(EW-01). 합산값은 로그 스케일링 후 정규화에 사용된다. Capping이 발생한 커밋은 조장에게 신호로 표시된다.
- **출처:** STR-2, L2, O-2, EW-01
- **수용기준:**
  - 단일 5,000줄 커밋 → 내부 집계 1,000.
  - 999줄 → 999 그대로.
  - Capping 발생 시 결과에 플래그 포함.
  - Capping이 발생한 모든 커밋이 조장에게 목록(작성자·커밋 식별·변경량)으로 표시된다.
- **시나리오:** SC-A 8단계, SC-B 1단계

#### FR-4.2b 비정상 빈도 신호 (EW-02)
- **무엇을:** 시스템은 특정 작성자의 단기 커밋 빈도가 평소의 3배를 초과하는 구간을 이상 신호로 식별하여 조장에게 표시한다.
- **출처:** STR-2, L2, EW-02
- **수용기준:**
  - 작성자의 평소 일평균 커밋 대비 3배 초과 일자가 존재 시 해당 작성자·기간이 신호 목록에 표시된다.
  - 신호 목록의 각 항목에 작성자·기간·해당 기간 커밋 수·평소 평균이 포함된다.
- **시나리오:** SC-B

#### FR-4.2c 이상 신호 예외 처리 (정상으로 표시)
- **무엇을:** 시스템은 표시된 이상 신호(Capping·EW-02 빈도·Z-Score 하위)를 조장이 개별적으로 *정상으로 표시*할 수 있게 하고, 정상 처리된 신호를 화면 표시에서 즉시 제외한다. 신호는 애초에 종합 점수에 반영되지 않으므로(STR-7) 본 기능은 *표시 전용 상태*만 변경하며 점수를 재산출하지 않는다. 정상 처리 이력은 현재 운영 세션 동안만 유지되고 영속화하지 않는다(OI-1).
- **출처:** STR-2, ConOps §7.2(검증 권유 성격), SC-B
- **수용기준:**
  - 각 이상 신호 카드에 "정상으로 표시" 동작이 제공된다.
  - 특정 신호를 정상으로 표시하면 해당 신호 항목이 신호 패널에서 즉시 사라진다.
  - 한 작성자의 동일 유형 신호가 복수일 때, 정상 처리는 (작성자·신호유형·근거식별자) 단위로 적용되어 나머지 동일 유형 신호는 유지된다.
  - 정상 처리는 종합 기여 지표(`total_score`)를 변경하지 않는다(신호는 점수 비반영, STR-7).
  - 새 분석을 수행하면 이전 세션의 정상 처리 이력은 초기화된다.
- **시나리오:** SC-B
- **비고(번호 정합):** 본 항목은 구 SRS의 `FR-4.2c(이상 신호 예외 처리)`를 정본화한 것이다. RR v1.3에서 삭제(UC-9)되었으나 코드(NormalizedSignalsTracker·AnomalySignalPanel)에 구현되어 복원하였다. 번호는 SRS 원체계(4.2c=예외, 4.2d=Z-Score)를 따르며, 이는 test-plan §1.3의 "Z-Score=FR-4.2d 통일" 결정과 정합한다(아래 FR-4.2d 헤더의 구 오기 4.2c→4.2d 정정 포함).

#### FR-4.2d Z-Score 하위 이상치 신호
- **무엇을:** 시스템은 정규화된 지표에서 Z-Score가 -1.5 이하인 항목이 2개 이상 존재하는 팀원을 *하위 이상치 신호*로 식별하여 조장에게 표시한다.
- **출처:** STR-2, SC-A 8단계
- **수용기준:**
  - 팀원의 Git·문서·메신저 Z-Score 중 -1.5 이하가 2개 이상 → 신호 표시.
  - 신호는 산점도(FR-5.1c)에서 해당 팀원의 점에 시각적 강조로 표시된다.
- **시나리오:** SC-A 8단계
- **비고(번호 정합):** 구 RR 본문은 본 신호를 `FR-4.2c`로 표기했으나 RTM·ConOps·architecture·test-plan §1.3은 일관되게 `FR-4.2d`로 사용해 왔다. 본 정정으로 전 문서를 **FR-4.2d=Z-Score**로 통일한다.

#### FR-4.3 데이터 소스 결측 시 동적 가중치 재조정
- **무엇을:** 임의의 데이터 소스(Git, 문서, 메신저) 중 일부가 부재하거나 분석 실패한 경우 시스템은 해당 소스의 가중치를 0으로 만들고, 나머지 가용한 소스의 가중치를 상대 비율을 유지하며 재조정한다. 재조정 후 가중치 합은 1.0(±0.0001)이다.
- **출처:** STR-1, STR-6, STR-8
- **수용기준:**
  - 메신저 입력 None → 예외 없이 종합 지표 반환, 가중치 합 1.0±0.0001.
  - Git 입력 None(저장소 부재) → 예외 없이 종합 지표 반환, 가중치 합 1.0±0.0001.
  - 문서 입력 None → 예외 없이 종합 지표 반환, 가중치 합 1.0±0.0001.
  - 3개 소스 중 1개만 가용 시 → 해당 소스 가중치 1.0, 종합 지표 = 해당 소스 정규화 점수.
  - 3개 소스 중 2개 가용 시 → 두 소스의 상대 비율 유지하며 합 1.0으로 재조정.
  - 동일 입력에 대해 데이터 소스 가용 조합별 결과가 결정론적으로 산출된다.
  - 3개 소스 모두 부재 → 분석 시작 단계에서 차단되며 "분석 가능한 데이터 소스가 없습니다." 안내 표시.

#### FR-4.4 가중치 프리셋·슬라이더
- **무엇을:** 조장은 3개 프리셋(개발 중심/기획 중심/균형 설정) 또는 슬라이더(0~1, 0.05 step)로 가중치를 조정한다. 가중치 슬라이더 조작 시 나머지 항목이 실시간으로 자동 연동된다. 합이 1.0이 아니면 분석 시작 불가.
- **출처:** STR-7
- **수용기준:**
  - "개발 중심" → 0.60 / 0.25 / 0.15.
  - "기획 중심" → 0.20 / 0.60 / 0.20.
  - "균형 설정" → 0.40 / 0.40 / 0.20.
  - 가중치 슬라이더를 조작할 때, 가중치 총합이 1.0이 되도록 나머지 항목이 자동으로 실시간 연동되어 변경된다.
  - 변경되는 현재 가중치 값이 슬라이더 옆에 숫자로 명확하게 표시된다.
  - 합 ≠ 1.0 → 분석 시작 버튼 비활성화 + 경고 "가중치 합계가 1.00이어야 합니다. 현재: X.XX".
  - 합 = 1.0 → 분석 시작 버튼 활성화.
  - (보조) 시스템은 한 축의 값을 조정할 때 나머지 두 축을 기존 비율대로 비례 재분배하여 합 1.0을 유지하는 재분배 연산을 제공할 수 있다. 임의 양수 입력은 합 1.0으로 정규화 가능하며, 현재 가중치가 어떤 프리셋과 일치하는지 역추적할 수 있다. 모든 연산은 결정론적이다(NFR-1.3).

### FR-5. 시각화·리포트

#### FR-5.1 차트 표시 (공통 규칙)
- **무엇을:** 시스템은 분석 완료 시 막대 차트, 레이더 차트, 산점도 3종을 표시한다. 세 차트는 독립 위젯 클래스로 분리 구현되며, 차트 간 연동은 Signal/Callback 경유로 한정한다.
- **출처:** STR-5, L5, O-3, SC-A 7~8단계
- **수용기준:**
  - 분석 미실행 상태에서 각 차트 패널에 "분석할 데이터가 없습니다." 안내 문구 표시.
  - 막대·레이더·산점도는 각각 별도의 ChartWidget 클래스로 구현되며, Widget 간 직접 참조 없이 Signal/Callback으로만 통신한다.
  - 분석 완료 트리거 시 세 차트가 동시 갱신된다.

#### FR-5.1a 막대 차트 (Bar Chart)
- **무엇을:** 시스템은 팀원별 종합 기여 지표를 막대 차트로 표시한다. hover 툴팁, 팀 평균선, 진입 애니메이션을 포함한다.
- **출처:** STR-5
- **수용기준:**
  - X축은 팀원명, Y축은 종합 기여 지표(0.0~1.0)이며 Y축 범위는 0.0~1.0으로 고정, 그리드 0.2 간격.
  - 막대 색상은 1위 강조색과 나머지 기본색으로 구분된다.
  - 막대 hover 시 툴팁이 표시되며 다음 6개 항목을 포함한다: 팀원명 / Git 정규화 점수(및 원시 추가 라인 수 등 구성요소) / 문서 정규화 점수(및 원시 글자수 등 구성요소) / 메신저 정규화 점수(및 원시 유효 발화수 등 구성요소) / 종합 기여 지표 / Capping 발동 여부.
  - 툴팁은 커서 우측 상단 오프셋으로 표시되며, 차트 경계를 벗어날 경우 좌측으로 자동 반전된다.
  - 팀 평균 수평 점선과 "팀 평균: X.XX" 레이블이 표시된다.
  - 팀 평균선의 Y값은 전체 팀원 종합 기여 지표의 산술평균(±0.0001)이다.
  - 각 막대 상단에 수치 레이블(소수점 2자리)이 표시된다.
  - 분석 완료 트리거 시 막대가 하단에서 최종 높이까지 상승하는 애니메이션이 재생된다(20 프레임, 30ms 간격).
  - 애니메이션 진행 중 hover 이벤트는 비활성, 완료 후 활성화된다.
  - 애니메이션 완료 후 막대 높이는 FR-4.3 종합 기여 지표 값과 ±0.001 이내로 일치한다.

#### FR-5.1b 레이더 차트 (Radar Chart)
- **무엇을:** 시스템은 팀원별 Git·문서·메신저 정규화 점수를 3축 레이더 차트로 표시한다. 꼭짓점 hover 툴팁, 범례 토글, 팀 평균 폴리곤 오버레이를 포함한다.
- **출처:** STR-5
- **수용기준:**
  - 꼭짓점 3개의 축 레이블은 "Git" / "문서" / "메신저"로 정확히 표시된다.
  - 동심원 그리드는 0.2 간격 5단계로 표시된다.
  - 팀원별 폴리곤이 각각 별도 색상으로 그려진다(최대 6명 자동 색상 배분).
  - 팀 평균 폴리곤이 회색 점선으로 오버레이되며 범례에 "팀 평균"으로 표시된다.
  - 팀원 N명 + 팀 평균 폴리곤이 표시되어 총 (N+1)개의 폴리곤이 렌더링된다.
  - 폴리곤 꼭짓점 hover 시 툴팁이 표시되며 다음 4개 항목을 포함한다: 팀원명 / 해당 지표명 / 정규화 점수 / 원시값(Git=추가 라인, 문서=글자수, 메신저=발화 수).
  - 범례 항목 클릭 시 해당 팀원의 폴리곤이 show/hide 토글된다.
  - 토글 상태는 bool 리스트로 관리되며 재분석 시 초기화된다.
  - 각 꼭짓점 방향으로 해당 지표 최고점 팀원명 레이블이 표시된다.
  - FR-4.3에 따라 메신저 데이터 결측 시 메신저 꼭짓점 축은 점선으로 처리되고 "(제외됨)" 레이블이 표시된다 (Git·문서 결측도 동일 규칙 적용).
  - 분석 완료 트리거 시 폴리곤이 중심에서 최종 형태로 확장되는 애니메이션이 재생된다(20 프레임, 30ms 간격, 팀원 순서대로 50ms 딜레이 순차 등장).
  - 애니메이션 완료 후 각 꼭짓점 반경은 해당 정규화 점수와 ±0.001 이내로 일치한다.

#### FR-5.1c 산점도 (Scatter Plot)
- **무엇을:** 시스템은 팀원별 (Git 정규화 점수 × 문서 정규화 점수) 분포를 산점도로 표시하고, 점 크기는 메신저 정규화 점수에 비례한다. 사분면 영역 어노테이션, 팀 평균 십자선, 레이더 차트와의 클릭 연동을 포함한다.
- **출처:** STR-5, SC-A 8단계
- **수용기준:**
  - X축은 "Git 점수"(0.0~1.0), Y축은 "문서 점수"(0.0~1.0)로 레이블된다.
  - 점 크기는 메신저 정규화 점수에 따라 40pt(점수 0.0)~200pt(점수 1.0)로 선형 매핑되며 매핑 정확도는 ±5pt이다.
  - FR-4.3에 따라 메신저 결측 시 점 크기는 80pt로 고정되고 테두리는 점선으로 처리된다.
  - 각 점 우측 상단에 팀원명 라벨이 표시된다.
  - 라벨 간 최소 거리가 30px 미만일 경우 라벨 위치가 상/하/좌/우 4방향 순으로 자동 조정되며, 4방향 모두 겹칠 경우 가장 덜 겹치는 방향이 선택된다.
  - 사분면 배경이 4분할 색상으로 표시되고 각 사분면 우측 하단에 영역 레이블이 표시된다.
    - X≥0.5, Y≥0.5 → 연초록 "올라운더"
    - X≥0.5, Y<0.5 → 연파랑 "개발 집중"
    - X<0.5, Y≥0.5 → 연주황 "문서 집중"
    - X<0.5, Y<0.5 → 연회색 "저참여"
  - 팀 평균 십자선(X 평균 수직 점선, Y 평균 수평 점선, 회색)이 표시되며, 교차점에 "팀 평균" 마커가 표시된다.
  - 십자선의 X값은 전체 팀원 Git 정규화 점수의 산술평균(±0.0001), Y값은 문서 정규화 점수의 산술평균(±0.0001)과 일치한다.

#### FR-5.7 결과 화면 계정 병합 (Post-hoc Account Merge)
- **무엇을:** 결과 화면에서 조장은 서로 다른 복수의 계정(사용자)을 하나의 팀원으로 병합할 수 있다. 한 팀원이 여러 계정·이메일·이름을 사용해 여러 인물로 분리 집계된 경우, 조장이 그 이름들을 선택해 1인으로 합치면 그래프가 단일 인격으로 갱신된다. 병합은 원시 지표 재집계→재정규화를 거친다.
- **출처:** STR-3, L3, SC-C, FR-1.3
- **수용기준:**
  - 결과에 표시된 모든 인물이 병합 컨트롤에 후보로 제시된다.
  - 조장이 복수 인물을 선택해 1인으로 병합하면, 그 인물들의 원시 지표가 합산되고 **전체 재정규화 후** 막대·레이더·산점도와 평균선·사분면이 모두 갱신된다.
  - 병합은 시각적 합산이 아니라 재집계 결과여야 한다(정규화는 팀원 집합 전체 의존, FR-4.1).
  - 매핑 대상을 하나도 선택하지 않은 상태에서는 확인(OK) 버튼이 비활성화되거나 눌리지 않아 데이터 분석을 아무것도 안 한 빈 화면이 출력되는 것을 방지한다.
  - 매핑 기능 중 취소(Cancel) 버튼을 누르면 전체 기능이 사라지지 않고 매핑만 취소되며 기존 화면 상태가 유지된다.
  - 병합 취소(분리)가 가능하며, 분리 시에도 동일하게 재집계된다.
  - 동일 병합 입력에 대해 결과가 결정론적으로 산출된다(NFR-1.3).
  - 병합 처리 중 중복 실행이 차단된다(NFR-1.2).
  - 시스템이 서로 다른 계정을 자동으로 병합하지 않는다(조장의 명시 선택만, FR-1.3).
- **시나리오:** SC-C 전체, SC-A 결과 검토 단계

### NFR-2. 보안·프라이버시

#### NFR-2.1 읽기 전용 입력 접근
- **무엇을:** 분석 대상 파일은 읽기 전용으로만 열린다. 리포트는 사용자가 명시 지정한 새 경로에만 쓰여진다.

- **무엇을:** 시스템은 외부 IP로 송신하지 않는다. 외부 링크는 OS 브라우저에 위임한다.

  - 코드 리뷰에서 네트워크 I/O 라이브러리(`requests`, `urllib`, `httpx`, `socket`, `http.client`) 임포트가 발견되면 Fail.

#### NFR-2.3 캐시 무결성·민감정보 비저장
- **무엇을:** 시스템은 정규화 점수·식별자·타임스탬프·가중치 설정만 캐시하며, 메시지 본문·소스코드 내용은 캐시하지 않는다. 캐시 쓰기는 원자적이며, 손상 시 자동 삭제 후 안내하고 초기 상태로 시작한다.
- **출처:** STR-9, C-2, C-8
- **수용기준:**
  - 캐시는 JSON으로 정상 역직렬화된다.
  - 캐시 내용에 메신저 본문이 포함되면 Fail.
  - 소스코드에서 `import pickle` / `from pickle` 발견 시 Fail.
  - 쓰기 중 강제 종료 후 재시작 → "캐시 파일이 손상되어 삭제되었습니다. 재분석이 필요합니다." 다이얼로그 표시 + 초기 화면 진입.
  - 정상 쓰기 완료 후 임시 파일이 디스크에 잔존하지 않는다.

#### NFR-2.4 캐시 단독 로드
- **무엇을:** 원본 파일이 부재한 상태에서도 캐시만으로 이전 분석 결과를 표시할 수 있다.

### NFR-3. 안정성

#### NFR-3.1 인코딩 자동 감지
- **무엇을:** 텍스트 파일 오픈 시 UTF-8 → CP949 순으로 시도하고, 둘 다 실패 시 해당 파일을 skip하며 에러 정보를 결과에 포함하되 앱은 계속 실행한다.
- **출처:** STR-6
- **수용기준:**
  - UTF-8 파일 → 1회 시도 성공.
  - CP949 파일 → 한글 무손상.
  - 미지원 인코딩 → 종료 없이 error 키 포함 결과 반환.

#### NFR-3.2 데이터 모듈 상호 격리
- **무엇을:** Git·OOXML·메신저 분석 모듈은 상호 결합도가 분리되어, 임의의 1개 또는 2개 모듈만 가용한 상황(나머지 부재·실패)에서도 가용한 모듈로 분석이 정상 동작한다.
- **출처:** STR-6, C-4
- **수용기준:**
  - 3개 모듈 중 임의의 1개를 삭제한 상태에서 나머지 2개 모듈의 단위 테스트가 ImportError 없이 통과한다(3가지 조합 모두).
  - 3개 모듈 중 임의의 1개가 RuntimeError를 던지도록 Mock한 상태에서, 나머지 2개 모듈의 결과가 최종 결과에 정상 포함된다(3가지 조합 모두).
  - 3개 모듈 중 1개만 가용한 상태에서도 분석이 정상 종료되며 FR-4.3 가중치 재조정이 적용된다.
  - 모듈 간 직접 import는 금지되며, 모든 통합은 상위 오케스트레이터를 경유한다.
  - pytest 종료 코드 0.

### NFR-4. 배포

#### NFR-4.1 실행 파일 아이콘 적용
- **무엇을:** 시스템 빌드 및 배포 시, 기본 실행 파일 아이콘이 아닌 QCE 전용 아이콘이 적용되어야 한다.
- **출처:** C-6
- **수용기준:**
  - 단일 실행 파일(.exe)에 지정된 앱 아이콘 파일(.ico)이 정상적으로 적용되어 표시된다.
