## 1. 문서 개요

### 1.1 목적과 문장 규약
본 문서는 Problem Statement가 정의한 문제와 ConOps가 정의한 운영 시나리오로부터 검증 가능한 요구사항을 도출하고, 그 추적성을 박제한다.

**문장 규약.** 본 문서의 모든 요구사항 문장은 *의무 사양*으로 해석된다. 표기상 한국어 종결형 "~한다"를 사용하되, 이는 모두 "~해야 한다"의 의미다. 수용기준은 Pass/Fail이 명확히 판정 가능한 형태로 글머리표로 분리하여 기술한다.

### 1.2 SRS와의 관계
프로젝트 작업 순서상 SRS v2.0이 본 문서보다 먼저 작성되었다. 따라서 본 문서의 작업은 창작이 아니라 정합성 감사다. SRS v2.0에 혼재된 구현 코드(라이브러리 호출·정규식 등)는 본 문서에서 제외하되, 사용자가 보는 텍스트·URL·임계값 등 사양적 값은 보존한다. ✏️ SRS는 본 문서의 결정을 반영한 v2.1로 동기화된다.

### 1.4 용어
- **종합 기여 지표(Aggregate Contribution Indicator):** 정규화된 Git·문서·메신저 지표를 가중치로 결합한 0~1 사이의 *참고용 수치*. 평가 결론이 아니며, 조장의 판단 근거다(CON-0).
- **미매핑 식별자(Unmapped Identifier):** 입력 소스에서 발견되었으나 조장이 어느 팀원에도 연결하지 않은 식별자. FR-1.2의 "Unknown"과 구분된다.
- **Unknown 작성자:** OOXML 메타데이터 자체가 비어 있는 문서의 작성자 분류. 단일 정의된 분류이며 미매핑 식별자와 다르다.
- **이상 신호(Anomaly Signal):** 시스템이 통계·규칙으로 식별하여 조장에게 표시하는 검토 권유. 판정·결론이 아니다.
- ✏️ **소통량(Communication Volume):** 산점도 X축에 사용되는 팀원별 메신저 기여 척도. 정규화된 유효 메시지 수(FR-3.3 불용어 사전 적용 후)로 정의한다.
- ✏️ **문서 작업 밀도(Document Work Density):** 산점도 Y축에 사용되는 팀원별 문서 기여 척도. 본 문서는 잠정적으로 *정규화된 문서 글자수*로 정의한다. 정식 정의는 OI-4로 미해결.

## 5. 기능 요구사항

### FR-1. 통합 입력 — 문서

#### FR-1.1 OOXML 텍스트·도형 추출
- **무엇을:** 시스템은 `.pptx`/`.docx` 파일에서 공백·개행을 제외한 유효 글자수와 도형 개수를 추출한다.
- **출처:** STR-1, L1, O-1
- **수용기준:**
  - 200자 .docx 입력 → 글자수 200±0.
  - 슬라이드 3장 × 박스 2 × 50자 .pptx → 글자수 300, 도형 수 6.
  - 빈 파일 → 글자수 0, 도형수 0, 예외 없음.
  - 손상된 파일 → 해당 파일 skip + 경고 메시지에 파일명 포함, 분석 계속.
- **시나리오:** SC-A 2단계

#### FR-1.2 작성자별 집계
- **무엇을:** 시스템은 OOXML 메타데이터의 작성자 식별자를 기준으로 글자수를 집계한다. 메타데이터가 부재한 파일은 "Unknown" 분류로 묶는다.
- **출처:** STR-1, L1
- **수용기준:**
  - Alice 100자 + Bob 150자 → `{"Alice":100,"Bob":150}`.
  - 메타 없음 → "Unknown" 키 존재, 예외 없음.
  - "Unknown"은 단일 정의된 분류이며, FR-1.3의 미매핑 식별자와는 다른 개념이다.
- **시나리오:** SC-A 2단계

#### FR-1.3 신원 매핑 (Global Alias Mapping)
- **무엇을:** 시스템은 모든 입력 소스에서 발견된 식별자를 조장에게 제시하고, 조장이 하나의 팀원에 여러 식별자를 수동 연결할 수 있게 한다(N:1).
- **출처:** STR-3, L3, O-1, SC-C
- **수용기준:**
  - Git·메신저·문서에서 추출된 모든 식별자가 매핑 UI에 빠짐없이 제시된다.
  - 한 팀원에 식별자 3개를 연결 시, 모든 결과 표시에서 그 셋의 기여가 단일 인격으로 합산된다.
  - 미매핑 식별자는 분석에서 제외된다.
  - 미매핑 식별자 경고에 각 식별자의 활동 규모(커밋 수·메시지 수 등)가 함께 표시된다.
  - 서로 다른 미매핑 식별자는 시스템에 의해 임의로 병합되지 않는다.
  - FR-1.2의 "Unknown" 분류와 미매핑 식별자는 별개로 처리된다.
- **시나리오:** SC-A 5단계, SC-C 전체

---

### FR-2. 통합 입력 — Git

#### FR-2.1 Git 로그 추출
- **무엇을:** 시스템은 로컬 Git 저장소에서 작성자별 커밋 수·추가 라인·삭제 라인을 수집한다. 잘못된 경로 입력 시 예외를 상위로 전파하지 않고 빈 결과를 반환한다.
- **출처:** STR-1, L1, O-1
- **수용기준:**
  - 3커밋(alice@test.com, +10/-5) 저장소 → 정확 일치.
  - 잘못된 경로 → 빈 결과, 예외 없음.
  - 10,000커밋 저장소 → 30초 이내 수집 완료.
- **시나리오:** SC-A 3단계

#### FR-2.2 Git 부재 시 안내
- **무엇을:** 메인 윈도우 표시 전 Git 가용성을 점검하고, 부재 시 모달 안내를 표시한 뒤 Git 기능만 비활성화한 채 앱을 계속 실행한다.
- **출처:** STR-1, ASM-1
- **수용기준:**
  - Git 미설치 환경 → 메인 윈도우 표시 전 모달 표시.
  - 모달에 "Git이 설치되어 있지 않거나 PATH에 등록되지 않았습니다." 문구를 포함한다.
  - 모달에 다운로드 링크 `https://git-scm.com/download/win`이 포함되고, 클릭 시 OS 기본 브라우저로 열린다.
  - 모달에 PATH 설정 안내 텍스트 1줄 이상이 포함된다.
  - 확인 버튼 클릭 → 앱 정상 실행, Git 기능만 비활성화.
  - Git 정상 설치 환경 → 모달 미표시.
- **시나리오:** SC-A 진입

### FR-3. 통합 입력 — 메신저

#### FR-3.1 카카오톡·슬랙 파싱
- **무엇을:** 시스템은 카카오톡 `.txt` 및 슬랙 `.csv` 내보내기 파일을 발화자·시각·메시지 구조로 정규화한다.
- **출처:** STR-1, L1
- **수용기준:**
  - 카카오톡 표준 형식 정확 파싱(발화자·메시지 분리).
  - 슬랙 3행 csv → 길이 3 리스트.
  - CP949 인코딩 카카오톡 파일 → 한글 무손상.
- **시나리오:** SC-A 4단계

#### FR-3.2 오염 줄 방어적 Skip
- **무엇을:** 시스템은 파싱 실패 줄을 카운트하며 건너뛰고, 정상 레코드와 skip 횟수를 함께 반환한다.
- **출처:** STR-1, STR-6
- **수용기준:**
  - 정상 10 + 오염 3 → records 10, skipped 3.
  - 전체 100줄 오염 → records 0, skipped 100, 예외 없음.
  - 모든 경우에 프로세스가 종료되지 않는다.

#### FR-3.3 불용어 사전 (Stopword Dictionary)
- **무엇을:** 시스템은 의미가 희박한 표현을 유효 메시지 집계에서 제외한다. 조장은 카테고리별 사전에 항목을 추가·삭제할 수 있다.
- **출처:** STR-4, L4, SC-D
- **수용기준:**
  - 시스템은 카테고리별 기본 불용어 사전을 내장한 상태로 시작한다(보수적 구성: 누가 봐도 의미 없는 표현에 한함).
  - 불용어 목록에 포함된 표현은 유효 메시지 수·유효 글자수 집계에서 제외된다.
  - 조장은 사전에 항목을 추가·삭제할 수 있다.
  - 사전 편집 결과는 조장이 '재분석'을 명시적으로 실행한 시점에만 집계에 반영된다.
  - 사전은 운영 세션을 넘어 로컬에 영속 저장된다.

**참고.** 불용어 사전은 조장이 의도적으로 입력한 *설정값*이며, 메신저 본문이 아니므로 NFR-2.3의 본문 비저장 정책과 충돌하지 않는다.

### FR-4. 분석 — 정규화·이상 신호·가중치

#### FR-4.1 Min-Max 정규화
- **무엇을:** 시스템은 이질적 단위 지표 3종(Git 라인·문서 글자·메신저 발화)을 0~1 척도로 정규화한다. 분산 0인 경우 전원 0.5. 결과는 소수점 4자리로 반올림한다.
- **출처:** STR-1, STR-8
- **수용기준:**
  - 입력 [0,50,100] → [0.0, 0.5, 1.0].
  - 입력 [75,75,75] → [0.5, 0.5, 0.5], ZeroDivisionError 없음.
  - 모든 결과 0.0 ≤ v ≤ 1.0.

#### FR-4.2 이상치 Capping과 신호 표시
- **무엇을:** 시스템은 단일 커밋의 추가 라인이 1,000줄을 초과할 경우 집계상 1,000줄로 제한한다(EW-01). 합산값은 로그 스케일링 후 정규화에 사용된다. Capping이 발생한 커밋은 조장에게 신호로 표시된다.
- **출처:** STR-2, L2, O-2, EW-01
- **수용기준:**
  - 단일 5,000줄 커밋 → 내부 집계 1,000.
  - 999줄 → 999 그대로.
  - Capping 발생 시 결과에 플래그 포함.
  - Capping이 발생한 모든 커밋이 조장에게 목록(작성자·커밋 식별·변경량)으로 표시된다.
- **시나리오:** SC-A 8단계, SC-B 1단계

#### FR-4.2b 비정상 빈도 신호 (EW-02)
- **무엇을:** 시스템은 특정 작성자의 단기 커밋 빈도가 평소의 3배를 초과하는 구간을 이상 신호로 식별하여 조장에게 표시한다. 점수를 자동으로 깎지 않는다.
- **출처:** STR-2, L2, EW-02
- **수용기준:**
  - 작성자의 평소 일평균 커밋 대비 3배 초과 일자가 존재 시 해당 작성자·기간이 신호 목록에 표시된다.
  - 신호 표시 자체는 종합 기여 지표 산출에 영향을 주지 않는다(CON-0).
- **시나리오:** SC-B

#### FR-4.2c 이상 신호 예외 처리 (UC-9)
- **무엇을:** 조장은 표시된 이상 신호(FR-4.2, FR-4.2b, FR-4.2d)를 *정상*으로 표시할 수 있다. 정상으로 표시된 신호는 종합 기여 지표 재계산에 반영된다.
- **출처:** STR-7, SC-B
- **수용기준:**
  - 조장이 신호를 '정상'으로 표시 시 해당 신호 표시는 제거된다.
  - 정상 처리 후 종합 기여 지표가 즉시 재계산되어 결과 표시에 반영된다.
  - 정상 처리 이력은 현재 운영 세션 동안 유지된다.
- **시나리오:** SC-B 3단계

#### ✏️ FR-4.2d Z-Score 하위 이상치 신호
- **무엇을:** 시스템은 정규화된 지표에서 Z-Score가 -1.5 이하인 항목이 2개 이상 존재하는 팀원을 *하위 이상치 신호*로 식별하여 조장에게 표시한다.
- **출처:** STR-2, SC-A 8단계
- **수용기준:**
  - 팀원의 Git·문서·메신저 Z-Score 중 -1.5 이하가 2개 이상 → 신호 표시.
  - 신호는 산점도(FR-5.1)에서 해당 팀원의 점에 시각적 강조로 표시된다.
  - 신호 표시 자체는 종합 기여 지표 산출에 영향을 주지 않는다(CON-0).
- **시나리오:** SC-A 8단계

#### FR-4.3 메신저 결측 시 동적 가중치 재조정
- **무엇을:** 메신저 데이터 부재 시 시스템은 메신저 가중치를 0으로 만들고, 나머지 가중치를 상대 비율을 유지하며 재조정한다. 재조정 후 가중치 합은 1.0(±0.0001)이다.
- **출처:** STR-1, STR-6
- **수용기준:**
  - 메신저 입력 None → 예외 없이 종합 지표 반환.
  - 재조정 후 가중치 합 1.0±0.0001.
  - 메신저 유/무 결과가 다르다.

#### FR-4.4 가중치 프리셋·슬라이더
- **무엇을:** 조장은 3개 프리셋(개발 중심/기획 중심/균형 설정) 또는 슬라이더(0~1, 0.05 step)로 가중치를 조정한다. 합이 1.0이 아니면 분석 시작 불가.
- **출처:** STR-7
- **수용기준:**
  - "개발 중심" → 0.60 / 0.25 / 0.15.
  - "기획 중심" → 0.20 / 0.60 / 0.20.
  - "균형 설정" → 0.40 / 0.40 / 0.20.
  - 합 ≠ 1.0 → 분석 시작 버튼 비활성화 + 경고 "가중치 합계가 1.00이어야 합니다. 현재: X.XX".
  - 합 = 1.0 → 분석 시작 버튼 활성화.

### FR-5. 시각화·리포트

#### ✏️ FR-5.1 차트 표시
- **무엇을:** 시스템은 분석 완료 시 다음 세 차트를 표시한다.
  - (a) 팀원별 막대 차트 — 종합 기여 지표(0~1).
  - (b) 팀원별 레이더 차트 — Git·문서·메신저 3축 정규화 점수.
  - (c) 팀원 산점도 — X축 소통량, Y축 문서 작업 밀도, 두 축 평균선 표시.
- **출처:** STR-5, L5, O-3, SC-A 7~8단계, SC-D
- **수용기준:**
  - 팀원 4명 → 막대 4개 + 각 막대에 팀원 레이블.
  - 레이더 차트 꼭짓점 레이블 "Git"·"문서"·"메신저" 표시.
  - 산점도에 팀원 수만큼의 점이 표시되고, 각 점에 팀원 레이블이 부착된다.
  - 산점도에 X축·Y축 평균선이 표시된다.
  - FR-4.2d 신호로 식별된 팀원의 점은 산점도에서 시각적으로 강조된다(붉은색 + 경고 오버레이).
  - 분석 미실행 상태에서 "분석을 실행하면 결과가 표시됩니다." 안내 문구 표시.

#### FR-5.2 리포트 저장 (.md/.csv)
- **무엇을:** 조장은 결과를 `.md` 또는 `.csv`로 사용자 지정 경로에 저장할 수 있다. CSV는 Excel 한글 호환을 위해 UTF-8 BOM으로 저장된다. 저장 성공 시 상태바에 3초간 알림.
- **출처:** STR-5, STR-10
- **수용기준:**
  - .md 저장 → 마크다운 테이블이 정상 렌더링.
  - .csv 저장 → Excel에서 한글 무손상.
  - 저장 성공 → 상태바에 경로 포함 메시지 3초 표시 후 사라짐.
  - .md/.csv 표 헤더에 "종합 지표"(또는 동의어)로 표기하며, "최종 평가" 등 판정 뉘앙스 표현을 사용하지 않는다(CON-0).

#### ✏️ FR-5.3 결측 데이터 경고
- **무엇을:** 메신저 파이프라인이 실패하거나 입력이 없는 경우 시스템은 UI 배너·md 블록쿼트·csv 행 모두에 동일한 경고 문구를 출력한다. 산점도는 메신저 결측 시 적절히 표시한다.
- **출처:** STR-1, STR-8
- **수용기준:**
  - 메신저 미입력 → UI에 노란 배경 배너 표시.
  - 배너 문구 "⚠ 메신저 데이터 형식 불일치로 인해 해당 지표가 평가에서 제외되었습니다."
  - 저장된 .md 파일 하단에 동일 문구가 블록쿼트로 포함.
  - 저장된 .csv 파일 하단에 빈 행 후 "WARNING" 행으로 동일 문구 포함.
  - 메신저 정상 입력 → 배너·경고 미포함.
  - 메신저 미입력 시 산점도의 X축은 "데이터 없음"으로 표기되거나 산점도 자체가 비활성 상태로 표시된다.

## 6. 비기능 요구사항

### NFR-1. UI 응답성·동시성

#### NFR-1.1 비동기 분석
- **무엇을:** 분석 연산은 메인 UI 스레드에서 분리되어 실행되며, UI는 응답 가능 상태를 유지한다.
- **출처:** STR-6
- **수용기준:**
  - 10만 줄 분석 중 메인 윈도우 드래그 가능(UI 프리징 없음).
  - 분석 시작 1초 이내 진행률 표시 출현.
  - 분석 완료·오류 시 진행률 사라짐, 결과 또는 오류 메시지 표시.

#### NFR-1.2 중복 실행 방지
- **무엇을:** 분석 실행 중 동일 트리거의 반복 입력은 단일 작업만 수행되도록 차단된다. 비정상 종료 후에도 트리거가 재활성화된다.
- **출처:** STR-6
- **수용기준:**
  - 분석 시작 0.5초 이내 분석 시작 버튼 비활성화.
  - 5회 연속 클릭 시 작업 1회만 실행.
  - 비정상 종료(RuntimeError 등) 후 버튼이 재활성화.

#### NFR-1.3 분석 결정론
- **무엇을:** 동일 입력과 동일 가중치·사전 설정에 대해 종합 기여 지표는 동일하게 산출된다.
- **출처:** STR-8
- **수용기준:**
  - 동일 입력·설정 2회 실행 → 결과 완전 일치.

### NFR-2. 보안·프라이버시

#### NFR-2.1 읽기 전용 입력 접근
- **무엇을:** 분석 대상 파일은 읽기 전용으로만 열린다. 리포트는 사용자가 명시 지정한 새 경로에만 쓰여진다.
- **출처:** STR-9, CON-1
- **수용기준:**
  - 원본 파일의 수정 타임스탬프가 분석 전후 동일.
  - 코드 리뷰에서 분석 대상 경로에 쓰기·이동·삭제 호출이 발견되면 Fail.

#### NFR-2.2 네트워크 차단
- **무엇을:** 시스템은 외부 IP로 송신하지 않는다. 외부 링크는 OS 브라우저에 위임한다.
- **출처:** CON-1
- **수용기준:**
  - 네트워크 모니터링 결과 QCE 프로세스의 외부 송신 0건.
  - 코드 리뷰에서 네트워크 I/O 라이브러리 임포트가 발견되면 Fail.

#### NFR-2.3 캐시 무결성·민감정보 비저장
- **무엇을:** 시스템은 정규화 점수·식별자·타임스탬프·가중치 설정만 캐시하며, 메시지 본문·소스코드 내용은 캐시하지 않는다. 캐시 쓰기는 원자적이며, 손상 시 자동 삭제 후 안내하고 초기 상태로 시작한다.
- **출처:** STR-9, CON-1, CON-7
- **수용기준:**
  - 캐시는 JSON으로 정상 역직렬화된다.
  - 캐시 내용에 메신저 본문이 포함되면 Fail.
  - 소스코드에서 `import pickle` / `from pickle` 발견 시 Fail.
  - 쓰기 중 강제 종료 후 재시작 → "캐시 파일이 손상되어 삭제되었습니다. 재분석이 필요합니다." 다이얼로그 표시 + 초기 화면 진입.
  - 정상 쓰기 완료 후 임시 파일이 디스크에 잔존하지 않는다.

#### NFR-2.4 캐시 단독 로드
- **무엇을:** 원본 파일이 부재한 상태에서도 캐시만으로 이전 분석 결과를 표시할 수 있다.
- **출처:** STR-6
- **수용기준:**
  - 원본 파일 전체 삭제 후 재시작 → 차트·점수 정상 표시.
  - 상태바에 "캐시 파일에서 이전 분석 결과를 불러왔습니다. (분석 일시: YYYY-MM-DD HH:MM)" 포함 메시지 표시.

### NFR-3. 안정성

#### NFR-3.1 인코딩 자동 감지
- **무엇을:** 텍스트 파일 오픈 시 UTF-8 → CP949 순으로 시도하고, 둘 다 실패 시 해당 파일을 skip하며 에러 정보를 결과에 포함하되 앱은 계속 실행한다.

  - CP949 파일 → 한글 무손상.
  - 미지원 인코딩 → 종료 없이 error 키 포함 결과 반환.

#### NFR-3.2 모듈 격리
- **무엇을:** 메신저 분석 모듈은 Git·OOXML 모듈로부터 결합도가 분리되어, 메신저 모듈 부재·실패 시에도 나머지 두 파이프라인이 정상 동작한다.
- **출처:** STR-6
- **수용기준:**
  - 메신저 모듈 삭제 후 Git·OOXML 단위 테스트 ImportError 없이 통과.
  - 메신저 모듈이 RuntimeError를 던지도록 Mock 시 Git·OOXML 결과가 최종 결과에 정상 포함.
  - pytest 종료 코드 0.
