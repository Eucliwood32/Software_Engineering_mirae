# 소프트웨어 요구사항 명세서 (SRS) - INVEST 개정본

**프로젝트명:** 팀 프로젝트 기여도 평가 도구 (QCE - Quantitative Contribution Evaluator)  
**버전:** v2.0 (INVEST 원칙 적용 개정본)  
**작성일자:** 2026년 5월 27일  
**작성 기준:** 각 요구사항은 Independent · Negotiable · Valuable · Estimable · Small · Testable 원칙을 준수하며, 개발자가 즉시 구현하고 QA가 독립적으로 검증할 수 있는 수준의 정량적 문장으로 기술한다.

## INVEST 원칙 적용 규칙

| 속성 | 적용 방식 |
| :--- | :--- |
| **I**ndependent | 각 요구사항은 다른 요구사항 없이도 단독으로 구현·테스트 가능하도록 범위를 분리한다. |
| **N**egotiable | 구현 방식(라이브러리, 알고리즘)은 협의 가능하나, 수용 기준(수치, 동작)은 고정한다. |
| **V**aluable | 각 항목이 조장(사용자)에게 전달하는 가치를 명시한다. |
| **E**stimable | 공수 산정이 가능하도록 범위를 단일 기능 단위로 한정한다. |
| **S**mall | 하나의 요구사항이 단일 책임(Single Responsibility)을 갖도록 분리한다. |
| **T**estable | 모든 요구사항에 Pass/Fail을 판정할 수 있는 수용 기준(Acceptance Criteria)을 명시한다. |

## 1. 프로젝트 개요

본 시스템(QCE)은 팀 프로젝트 조장이 조원의 실제 기여도를 정량적으로 평가할 수 있도록 지원하는 **Windows 전용 로컬 데스크톱 애플리케이션**이다. Git 커밋 이력, OOXML 문서 메타데이터, 메신저 대화 로그 세 가지 소스를 로컬에서 파싱·집계하여 팀원별 기여도 점수를 산출하고 시각화한다. 외부 네트워크 연결이나 클라우드 연동을 완전히 배제한다.

| :--- | :--- | :--- |

### FR-1: OOXML 문서 분석 파이프라인

#### FR-1.1 — .pptx / .docx 파일 파싱 및 글자 수 추출

> **가치:** 조장이 문서 기여량을 자동 집계하여 수작업 없이 팀원별 작성 분량을 파악할 수 있다.

**구현 기준(How):**
- `python-pptx`, `python-docx` 라이브러리를 사용하여 `.pptx`와 `.docx` 파일의 XML 구조를 직접 파싱한다.
- `.pptx`의 경우 각 슬라이드(`prs.slides`)의 모든 `shape.text_frame.text`를 순회하여 텍스트를 추출한다.
- `.docx`의 경우 `document.paragraphs`를 순회하여 `paragraph.text`를 추출한다.
- 추출한 텍스트에서 공백·개행 문자를 제외한 순수 글자 수(`len(text.replace(' ', '').replace('\n', ''))`)를 계산한다.
- 슬라이드 또는 단락 단위로 추가된 도형(shape) 개수도 별도 카운트한다.

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | `python-pptx`, `python-docx` 라이브러리로 파일을 열고 XML 구조를 순회하여 텍스트·도형 원시 데이터를 추출한다. |
| **프론트엔드** | 분석할 파일을 선택하는 파일 선택 다이얼로그 UI를 제공하고, 파싱 진행 상태(Progress Bar)를 표시한다. |
| **비즈니스 로직** | 추출된 원시 텍스트에서 공백·개행을 제거한 순수 글자 수를 계산하고, 도형 개수를 카운트하는 집계 로직을 구현한다. |

**수용 기준(Acceptance Criteria):**
1. 텍스트 200자가 포함된 테스트용 `.docx` 파일을 입력했을 때, 시스템이 출력하는 글자 수가 `200 ± 0`자로 정확히 일치한다.
2. 슬라이드 3장, 각 슬라이드에 텍스트 박스 2개(각 50자)가 있는 `.pptx`를 파싱하면 총 글자 수 `300`자, 도형 수 `6`개가 출력된다.
3. 빈 파일(내용 없음) 입력 시 글자 수 `0`, 도형 수 `0`을 반환하고 예외(Exception)가 발생하지 않는다.

#### FR-1.2 — 팀원 식별자 기반 기여 데이터 재정렬

> **가치:** 파일 단위가 아닌 팀원 단위로 데이터를 집계하여 "누가 얼마나 썼는지"를 즉시 확인할 수 있다.

**구현 기준(How):**
- 파싱 결과를 `Dict[str, int]` 구조(`{"팀원이름": 총_글자수}`)로 집계한다.
- 작성자 식별자는 `.docx`의 경우 `core_properties.author`, `.pptx`의 경우 `core_properties.last_modified_by` 필드에서 읽는다.
- 식별자가 비어있는(`""` 또는 `None`) 경우 해당 파일의 기여자를 `"Unknown"` 키로 분류한다.

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | `.docx`의 `core_properties.author`, `.pptx`의 `core_properties.last_modified_by` 메타데이터 필드를 읽어 팀원 식별자를 추출한다. |
| **프론트엔드** | 해당 없음 (내부 데이터 재정렬 단계). |
| **비즈니스 로직** | 파싱 결과를 `Dict[str, int]` 구조로 팀원별 합산 집계하고, 식별자가 없는 경우 `"Unknown"` 키로 분류하는 로직을 구현한다. |

**수용 기준(Acceptance Criteria):**
1. 작성자 `"Alice"`가 100자, `"Bob"`이 150자를 작성한 `.docx` 파일을 입력하면 집계 결과가 `{"Alice": 100, "Bob": 150}`으로 출력된다.
2. 작성자 메타데이터가 없는 파일 입력 시 결과 딕셔너리에 `"Unknown"` 키가 존재하고 예외가 발생하지 않는다.

### FR-2: 로컬 Git 저장소 분석 파이프라인

#### FR-2.1 — git log 기반 커밋 데이터 추출

> **가치:** 조장이 코드 기여량을 자동으로 집계하여 커밋 수·추가/삭제 라인 수를 팀원별로 확인할 수 있다.

**구현 기준(How):**
- `subprocess.run(["git", "log", "--numstat", "--format=%H|%ae|%ai"], cwd=repo_path, capture_output=True, text=True, timeout=30)` 형태로 git을 호출한다.
- 파싱 결과를 다음 스키마로 구성한다: `{"author_email": {"commits": int, "additions": int, "deletions": int}}`.
- `subprocess.CalledProcessError` 및 `FileNotFoundError` 발생 시 빈 딕셔너리(`{}`)를 반환하고 예외를 상위로 전파하지 않는다.
- git 명령 타임아웃은 **30초**로 고정한다.

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | `subprocess.run()`으로 `git log --numstat` 명령을 호출하고, stdout 문자열을 줄 단위로 읽어 커밋 해시·이메일·타임스탬프·추가/삭제 라인 수를 파싱하여 반환한다. |
| **프론트엔드** | 분석할 로컬 Git 저장소 경로를 선택하는 폴더 선택 다이얼로그를 제공한다. |
| **비즈니스 로직** | 파싱된 커밋 로그를 author_email 기준으로 집계하여 `{"author_email": {"commits": int, "additions": int, "deletions": int}}` 스키마로 구성하는 집계 함수를 구현한다. |

**수용 기준(Acceptance Criteria):**
1. 커밋 3개(작성자: `alice@test.com`, 추가 10줄·삭제 5줄)가 있는 테스트 저장소를 입력하면 결과가 `{"alice@test.com": {"commits": 3, "additions": 10, "deletions": 5}}`와 일치한다.
2. 유효하지 않은 경로(예: `C:\not_a_repo`)를 입력하면 예외가 발생하지 않고 빈 딕셔너리(`{}`)를 반환한다.
3. 대용량 저장소(커밋 10,000개 기준)에서 git log 호출이 **30초 이내**에 완료된다.

#### FR-2.2 — 앱 시작 시 Git 설치 여부 헬스체크 및 안내 팝업

> **가치:** Git 미설치 상태에서 앱이 조용히 실패하는 대신, 조장이 문제 원인을 즉시 파악하고 해결할 수 있다.

**구현 기준(How):**
- 애플리케이션 메인 윈도우가 표시되기 **전**, `subprocess.run(["git", "--version"], capture_output=True, timeout=5)` 호출을 1회 수행한다.
- `FileNotFoundError` 또는 반환 코드 `!= 0` 발생 시 다음을 포함한 모달 팝업(Modal Dialog)을 즉시 표시한다:
  - 경고 문구: `"Git이 설치되어 있지 않거나 PATH에 등록되지 않았습니다."`
  - Git 공식 다운로드 링크: `https://git-scm.com/download/win` (클릭 시 기본 브라우저로 열기)
  - 환경변수 PATH 설정 방법 안내 텍스트 1줄 이상
  - `[확인]` 버튼 (클릭 시 팝업 닫고 Git 분석 기능만 비활성화한 채로 앱 계속 실행)

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | 앱 시작 시 `subprocess.run(["git", "--version"])` 호출 결과를 확인하고, 실패 시 프론트엔드로 오류 Signal을 전달한다. |
| **프론트엔드** | Git 미설치 시 경고 문구·다운로드 링크·PATH 설정 안내를 포함한 모달 팝업을 표시하고, `[확인]` 클릭 시 Git 분석 관련 UI 버튼을 비활성화한다. |
| **비즈니스 로직** | 해당 없음 (시스템 환경 검사 단계). |

**수용 기준(Acceptance Criteria):**
1. Git이 PATH에 없는 환경에서 앱 실행 시, 메인 윈도우 표시 전에 위 내용을 포함한 팝업이 표시된다.
2. 팝업의 `[확인]` 버튼 클릭 후 앱이 종료되지 않고 메인 윈도우가 정상 표시된다.
3. Git 정상 설치 환경에서는 팝업이 표시되지 않는다.

### FR-3: 메신저 대화 로그 분석 파이프라인

#### FR-3.1 — 카카오톡·슬랙 내보내기 파일 파싱

> **가치:** 조장이 별도 가공 없이 메신저에서 내보낸 파일을 그대로 넣어 발화량을 팀원별로 집계할 수 있다.

**구현 기준(How):**
- **카카오톡 `.txt` 포맷** 파싱 패턴: 정규식 `r"^\d{4}년 \d{1,2}월 \d{1,2}일"` (날짜 구분선), `r"^(.+?)\s:\s(.+)$"` (발화자·메시지 분리)를 적용한다.
- **슬랙 `.csv` 포맷** 파싱: `csv.DictReader`로 읽되 헤더 행의 `user`, `text`, `timestamp` 컬럼을 필수로 사용한다.
- 파싱 결과를 `List[Dict]` 형태(`[{"author": str, "timestamp": str, "message": str}]`)로 반환한다.
- 파일 인코딩은 `UTF-8` 우선 시도 후 실패 시 `CP949(EUC-KR)`로 재시도한다(NFR-3.1 참조).

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | 파일을 `UTF-8` → `CP949` 순서로 열고, 카카오톡 정규식 패턴 또는 `csv.DictReader`로 줄 단위 파싱을 수행하여 `List[Dict]` 형태로 반환한다. |
| **프론트엔드** | 메신저 로그 파일(`.txt` / `.csv`) 선택 다이얼로그를 제공하고, 파싱 완료 후 파싱된 발화 수를 상태바에 표시한다. |
| **비즈니스 로직** | 해당 없음 (원시 파싱 단계, 집계는 FR-4에서 수행). |

**수용 기준(Acceptance Criteria):**
1. `"2024년 1월 5일\nAlice : 안녕하세요\nBob : 네 안녕하세요"` 형식의 카카오톡 `.txt` 파일 파싱 결과가 `[{"author": "Alice", "message": "안녕하세요", ...}, {"author": "Bob", "message": "네 안녕하세요", ...}]` 와 일치한다.
2. `user,text,timestamp` 헤더를 가진 슬랙 CSV 3행 파일 파싱 시 결과 리스트의 길이가 `3`이다.
3. CP949 인코딩으로 저장된 카카오톡 파일 파싱 시 한글이 깨지지 않는다.

#### FR-3.2 — 오염된 줄(Line) 건너뛰기(Skip) 방어적 파싱

> **가치:** 플랫폼 업데이트로 포맷이 일부 변경되어도 앱이 크래시 없이 가용한 데이터를 최대한 추출한다.

**구현 기준(How):**
- 파일을 줄 단위(`for line in file`)로 순회하며, 각 줄 처리를 `try-except Exception` 블록으로 감싼다.
- 파싱 패턴과 불일치하거나 예외가 발생한 줄은 `skip_count` 카운터를 1 증가시키고 `continue`로 건너뛴다.
- 함수 반환값에 `skip_count`를 포함한다: `{"records": List[Dict], "skipped_lines": int}`.

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | `for line in file` 스트리밍 순회 중 각 줄을 `try-except`로 감싸고, 파싱 실패 줄은 `skip_count`를 증가시키며 `continue`로 건너뛴다. 결과를 `{"records": ..., "skipped_lines": int}` 형태로 반환한다. |
| **프론트엔드** | 해당 없음 (내부 파싱 로직 단계). |
| **비즈니스 로직** | `skip_count`가 1 이상인 경우 상위 오케스트레이터에 경고 플래그(`has_skipped_lines: True`)를 포함하여 FR-5.3 경고 배너 표시 여부를 결정한다. |

**수용 기준(Acceptance Criteria):**
1. 정상 줄 10개 + 오염된 줄(예: `"@@@INVALID@@@"`) 3개가 혼재된 파일 파싱 시, `records` 리스트 길이 `10`, `skipped_lines` 값 `3`이 반환된다.
2. 파일 전체가 오염된 경우(100줄 모두 파싱 불가) `records`가 빈 리스트, `skipped_lines`가 `100`이 반환되고 예외가 발생하지 않는다.
3. 위 두 경우 모두 함수 호출이 정상 반환(return)되고 앱 프로세스가 종료되지 않는다.

### FR-4: 최종 기여도 특성 매트릭스 및 산출 알고리즘

#### FR-4.1 — 지표 Min-Max 정규화

> **가치:** 단위가 다른 이질적인 지표(라인 수, 글자 수, 발화 횟수)를 0~1 척도로 통합하여 공정하게 비교할 수 있다.

**구현 기준(How):**
- 정규화 공식: `normalized = (x - min) / (max - min)`. 단, `max == min`(모든 팀원 값 동일)인 경우 전원 `0.5`를 반환한다.
- 정규화 대상 지표 3종: Git 추가 라인 수, OOXML 텍스트 글자 수, 메신저 발화 횟수.
- 정규화 결과값은 소수점 4자리까지 `round(..., 4)`로 반올림한다.

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | 해당 없음 (순수 연산 단계). |
| **프론트엔드** | 해당 없음 (순수 연산 단계). |
| **비즈니스 로직** | `normalize(values: List[float]) -> List[float]` 함수를 구현한다. `max == min` 예외 처리, `round(..., 4)` 반올림, 결과 범위 보장(`0.0 ≤ v ≤ 1.0`)을 포함한다. |

**수용 기준(Acceptance Criteria):**
1. 입력 `[0, 50, 100]`에 대해 정규화 결과가 정확히 `[0.0, 0.5, 1.0]`이다.
2. 입력 `[75, 75, 75]`(모든 값 동일)에 대해 결과가 `[0.5, 0.5, 0.5]`이고 `ZeroDivisionError`가 발생하지 않는다.
3. 반환된 모든 값이 `0.0 ≤ v ≤ 1.0` 범위 내에 있다.

#### FR-4.2 — 이상치 Capping 및 로그 스케일링

> **가치:** 대용량 코드 복사·붙여넣기 등 비정상 기여로 인한 점수 왜곡을 방지하여 평가의 신뢰도를 높인다.

**구현 기준(How):**
- **Git 추가 라인 수 Capping 임계치:** 단일 커밋의 추가 라인 수가 `1,000줄`을 초과하면, 해당 커밋의 추가 라인 수를 `1,000`으로 강제 고정(Capping)한다.
- **로그 스케일 적용:** Capping 이후 전체 팀원 합산 추가 라인 수에 `math.log1p(x)` (밑 e의 자연로그, 0 입력 보호)를 적용한 뒤 정규화한다.
- Capping이 적용된 커밋이 1개 이상 존재하는 경우 `capping_applied: True` 플래그를 반환값에 포함한다.

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | 해당 없음 (순수 연산 단계). |
| **프론트엔드** | Capping이 적용된 경우(`capping_applied: True`) 대시보드에 `"일부 커밋이 이상치 처리되었습니다."` 안내 문구를 표시한다. |
| **비즈니스 로직** | 커밋별 추가 라인 수가 1,000을 초과하면 1,000으로 Capping하고, 팀원 합산 후 `math.log1p(x)` 로그 스케일링을 적용하는 전처리 함수를 구현한다. `capping_applied` 플래그를 반환값에 포함한다. |

**수용 기준(Acceptance Criteria):**
1. 단일 커밋에 추가 라인 수 `5,000`을 입력하면 내부 집계에 사용되는 값이 `1,000`이다.
2. 추가 라인 수 `999`를 입력하면 Capping 없이 원본값 `999`가 사용된다.
3. Capping 적용 시 반환 딕셔너리에 `"capping_applied": True`가 포함된다.

#### FR-4.3 — 메신저 파이프라인 결측 시 동적 가중치 재조정

> **가치:** 메신저 데이터가 없어도 앱이 Git·OOXML 데이터로 기여도를 정상 산출하여 평가를 중단 없이 완료할 수 있다.

**구현 기준(How):**
- 기본 가중치 예시: `W_git = 0.4, W_doc = 0.4, W_msg = 0.2` (FR-4.4 프리셋 기준).
- 메신저 데이터 부재 시 `W_msg = 0.0`으로 설정하고, 나머지 가중치를 **상대 비율 유지 스케일링**으로 재조정한다.
  - 공식: `W_i_new = W_i / (1.0 - W_msg)` (단, `W_i`는 메신저를 제외한 각 지표의 원래 가중치)
  - 재조정 결과: `W_git_new = 0.5, W_doc_new = 0.5`
- 재조정 후 모든 가중치의 합이 `round(sum, 4) == 1.0`임을 내부에서 assert로 검증한다.
- 최종 기여도 점수 = `Σ (normalized_i × W_i_new)` 이며, 결과값을 소수점 4자리로 반올림한다.

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | 메신저 파이프라인의 성공·실패 여부를 상위 오케스트레이터에 반환하여 결측 여부를 알린다. |
| **프론트엔드** | 해당 없음 (내부 알고리즘 단계). |
| **비즈니스 로직** | `calculate_score(git, doc, msg, weights)` 함수에서 `msg`가 `None`인 경우 `W_i_new = W_i / (1.0 - W_msg)` 공식으로 가중치를 동적 재조정하고, 재조정 후 합계 assert를 수행하는 로직을 구현한다. |

**수용 기준(Acceptance Criteria):**
1. 메신저 입력 파일이 `None`일 때 함수가 예외 없이 기여도 점수 딕셔너리를 반환한다.
2. 동 조건에서 내부적으로 사용된 가중치 합이 `1.0`(± 0.0001 허용)이다.
3. 메신저 데이터가 정상 제공될 때와 부재 시 각각의 출력값이 **다른** 점수를 반환한다(재조정 동작 확인).

| **비즈니스 로직** | 오케스트레이터에서 메신저 파이프라인 실패 여부를 `msg_missing: bool` 플래그로 집계하여 프론트엔드와 백엔드 저장 로직에 전달한다. |

**수용 기준(Acceptance Criteria):**
1. 메신저 파일 미입력 상태로 분석 실행 시, 대시보드 하단에 노란 배경의 경고 배너가 표시된다.
2. 동 조건에서 저장된 `.md` 파일에 위 경고 블록쿼트 문자열이 포함된다.
3. 메신저 데이터가 정상 입력된 경우 경고 배너와 경고 텍스트가 표시·포함되지 않는다.

### NFR-1: UI 비동기 처리 (UI Thread Safety)

> **가치:** 대용량 파일 분석 중에도 UI가 응답 가능한 상태를 유지하여 사용자 경험을 보호한다.

**구현 기준(How):**
- 파싱·분석 연산 전체를 메인 UI 스레드와 분리된 **Worker Thread** (예: `QThread` + `QRunnable`, `threading.Thread`)에서 실행한다.
- Worker Thread 시작 시 UI에 **진행률 표시줄(Progress Bar)**을 표시하고, 완료 또는 오류 발생 시 Signal/Callback으로 메인 스레드에 통보하여 UI를 업데이트한다.
- Worker Thread 실행 중 메인 스레드에서 창 이동, 버튼 클릭(분석 시작 제외) 등의 UI 이벤트가 처리 가능해야 한다.

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | 파싱·분석 연산을 Worker Thread 내에서 실행하고, 완료·오류 시 Signal로 결과를 메인 스레드에 전달한다. |
| **프론트엔드** | Worker Thread 시작 시 Progress Bar를 표시하고, 완료·오류 Signal 수신 시 Progress Bar를 제거하고 결과 또는 오류 메시지를 업데이트한다. |
| **비즈니스 로직** | 해당 없음 (스레딩 아키텍처 단계). |

**수용 기준(Acceptance Criteria):**
1. 10만 줄 이상의 `.txt` 파일을 분석하는 동안 메인 윈도우를 드래그하면 창이 마우스를 따라 이동한다(UI 프리징 없음).
2. 분석 시작 후 1초 이내에 진행률 표시줄이 UI에 나타난다.
3. 분석 완료 또는 오류 발생 시 진행률 표시줄이 사라지고 결과 또는 오류 메시지가 UI에 표시된다.

#### NFR-1.2 — 분석 중복 실행 방지 (Race Condition Guard)

> **가치:** 사용자가 `[분석 시작]`을 연속으로 클릭해도 Worker Thread가 중복 실행되지 않아 기여도 점수가 2배로 누적되는 데이터 오염을 방지한다.

**구현 기준(How):**
- 애플리케이션 전역 상태에 `is_analyzing: bool = False` 플래그를 선언한다.
- `[분석 시작]` 버튼 클릭 핸들러 진입 시 `is_analyzing`이 `True`이면 즉시 `return`하여 새 Worker Thread 생성을 차단한다.
- Worker Thread **시작 직전** `is_analyzing = True`로 설정하고, `[분석 시작]` 버튼을 `setEnabled(False)`로 비활성화한다.
- Worker Thread **종료 시**(성공·오류·사용자 취소 구분 없이) `is_analyzing = False`로 재설정하고, `[분석 시작]` 버튼을 `setEnabled(True)`로 복원한다.
- Thread 종료 후 상태 복원은 반드시 메인 스레드에서 Signal/Callback으로 처리하며, Worker Thread 내부에서 직접 UI 위젯을 수정하지 않는다.

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | Worker Thread 시작·종료 이벤트를 Signal로 메인 스레드에 발행하고, `try/finally`로 오류 시에도 종료 Signal이 반드시 전달되도록 보장한다. |
| **프론트엔드** | `is_analyzing` 플래그 및 `[분석 시작]` 버튼 `setEnabled()` 제어, Signal 수신 후 버튼 상태 복원 UI 로직을 구현한다. |
| **비즈니스 로직** | 해당 없음 (UI 상태 관리 단계). |

**수용 기준(Acceptance Criteria):**
1. 분석 시작 후 **0.5초 이내**에 `[분석 시작]` 버튼이 비활성화(disabled) 상태로 전환된다.
2. 분석 실행 중 `[분석 시작]` 버튼을 5회 연속 클릭해도 Worker Thread는 1개만 실행된다. (작업관리자 또는 `threading.active_count()` 로그로 검증)
3. 분석 완료 후 동일 입력으로 재분석 시 최종 기여도 점수가 첫 실행 결과와 동일하다(중복 누적 없음).
4. Worker Thread가 `RuntimeError`로 비정상 종료된 경우에도 `[분석 시작]` 버튼이 다시 활성화된다.

#### NFR-2.3 — 익명화 통계 캐시 파일 저장 (원자적 쓰기 포함)

**구현 기준(How):**
- 분석 완료 후 팀원별 정규화 점수 딕셔너리(`Dict[str, Dict[str, float]]`)를 반드시 `json.dumps()`**만** 사용하여 직렬화한다. `pickle` 모듈은 소스코드 전체에서 `import`를 금지한다.
- 캐시 파일에 저장 가능한 데이터 유형: 팀원 식별자(이름·이메일), 정규화된 수치 점수, 분석 타임스탬프, 가중치 설정값.
- 캐시 파일에 저장 **금지** 데이터: 원본 메시지 텍스트(`str` 타입의 메신저 메시지 본문), 소스코드 파일 내용.
- **원자적 쓰기(Atomic Write):** 캐시 저장은 아래 3단계 순서로만 수행한다.
  1. 최종 경로와 동일한 디렉터리에 임시 파일(`.qce_cache.tmp`)에 JSON 내용을 완전히 쓴다.
  2. `tmp_file.flush()` 및 `os.fsync(tmp_file.fileno())`로 디스크 기록을 보장한다.
  3. `os.replace('.qce_cache.tmp', '.qce_cache')`로 단일 원자 연산으로 교체한다.
- **손상 캐시 복구:** 앱 시작 시 `.qce_cache` 로드 중 `json.JSONDecodeError` 또는 `KeyError`가 발생하면:
  1. 손상된 캐시 파일을 즉시 삭제(`os.remove`)한다.
  2. 모달 다이얼로그로 `"캐시 파일이 손상되어 삭제되었습니다. 재분석이 필요합니다."` 메시지를 표시한다.
  3. 빈 초기 상태(분석 결과 없음)로 앱을 정상 시작한다.

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | `.qce_cache.tmp` 임시 파일 쓰기 → `fsync()` → `os.replace()` 원자 교체, 손상 시 파일 삭제 및 초기화 로직을 구현한다. |
| **프론트엔드** | 캐시 손상 감지 시 `"캐시 파일이 손상되어 삭제되었습니다. 재분석이 필요합니다."` 모달 다이얼로그를 표시한다. |
| **비즈니스 로직** | 캐시에 저장할 데이터 스키마(`Dict[str, Dict[str, float]]`)를 정의하고, 원본 텍스트 데이터 포함 여부를 검증하는 함수를 구현한다. |

**수용 기준(Acceptance Criteria):**
1. 분석 완료 후 생성된 `.qce_cache` 파일을 `json.loads()`로 역직렬화하면 팀원별 수치 점수 딕셔너리가 정상 복원된다.
2. 역직렬화된 객체를 재귀적으로 순회하여 메신저 원본 메시지 문자열(분석에 사용한 입력 텍스트)이 포함되어 있으면 해당 테스트는 Fail이다.
3. 소스코드 전체에서 `import pickle` 또는 `from pickle` 구문이 발견되면 코드 리뷰 Fail로 판정한다.
4. 쓰기 도중 프로세스를 강제 종료(`taskkill /F`)한 뒤 앱을 재시작하면, 예외가 발생하지 않고 `"캐시 파일이 손상되어 삭제되었습니다."` 다이얼로그가 표시된 후 초기 화면이 열린다.
5. 정상 쓰기 완료 후 `.qce_cache.tmp` 임시 파일이 디스크에 남아있지 않다.

#### NFR-2.4 — 원본 파일 없이 캐시만으로 대시보드 로드

2. 동 조건에서 상태바에 `"캐시 파일에서 이전 분석 결과를 불러왔습니다."` 텍스트가 포함된 메시지가 표시된다.

### NFR-3: 안정성 및 테스트 가능성

#### NFR-3.1 — 파일 인코딩 자동 감지 및 방어적 예외 처리

**구현 기준(How):**
- 모든 텍스트 파일 오픈 시 아래 순서로 인코딩을 시도한다:
  1. `open(path, 'r', encoding='utf-8')`
  2. `UnicodeDecodeError` 발생 시 `open(path, 'r', encoding='cp949')` 재시도
  3. 두 번째도 실패하면 해당 파일을 건너뛰고 `{"error": "encoding_failed", "path": path}` 딕셔너리를 반환한다.
- 세 번째 단계 이후 앱은 계속 실행된다.

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | 파일 오픈 시 `UTF-8` → `CP949` 순서로 재시도하는 `safe_open(path)` 함수를 구현하고, 두 번 모두 실패 시 `{"error": "encoding_failed", "path": path}`를 반환한다. |
| **프론트엔드** | 인코딩 실패로 파일이 건너뛰어진 경우 상태바 또는 로그 패널에 해당 파일 경로와 함께 `"인코딩 감지 실패로 건너뜀"` 메시지를 표시한다. |
| **비즈니스 로직** | 해당 없음 (파일 읽기 인프라 단계). |

**수용 기준(Acceptance Criteria):**
1. UTF-8 인코딩 파일을 입력하면 1회 시도 만에 정상 파싱된다.
2. CP949 인코딩 파일을 입력하면 UTF-8 실패 후 CP949로 재시도하여 한글이 깨지지 않고 파싱된다.
3. 지원하지 않는 인코딩(예: Shift-JIS) 파일 입력 시 앱이 종료되지 않고 `error` 키를 포함한 딕셔너리가 반환된다.

---

#### NFR-3.2 — 메신저 모듈 결합도 완화 검증 (격리 테스트)

**구현 기준(How):**
- `messenger_parser` 모듈을 독립된 함수/클래스로 분리하여, Git 분석 모듈(`git_parser`)과 OOXML 분석 모듈(`ooxml_parser`)이 `messenger_parser`를 직접 `import`하지 않도록 설계한다.
- 메신저 분석 결과는 공통 인터페이스(`Dict[str, int]` 형태)로 전달되며, 상위 오케스트레이터(`analyzer.py` 등)가 결과를 통합한다.

**담당 역할 (Role Assignment):**

| 역할 | 담당 내용 |
| :--- | :--- |
| **백엔드** | `git_parser`, `ooxml_parser`, `messenger_parser`를 독립된 모듈로 분리하고, 상위 `analyzer.py`(오케스트레이터)가 결과를 통합하는 구조로 설계한다. |
| **프론트엔드** | 해당 없음 (모듈 아키텍처 단계). |
| **비즈니스 로직** | 각 파서의 반환 타입을 `Dict[str, int]` 공통 인터페이스로 통일하고, 오케스트레이터가 이를 수신하여 최종 점수를 산출하는 파이프라인을 구현한다. |

**수용 기준(Acceptance Criteria):**
1. `messenger_parser.py` 파일을 프로젝트에서 삭제(또는 파일명 변경)한 상태에서 `git_parser`와 `ooxml_parser` 단독 단위 테스트(`pytest`)가 `ImportError` 없이 모두 통과한다.
2. `messenger_parser`가 `RuntimeError`를 발생시키도록 Mocking한 통합 테스트에서, Git 분석 및 OOXML 분석 결과가 정상값으로 포함된 최종 결과 딕셔너리가 반환된다.
3. 위 통합 테스트에서 `pytest` 종료 코드가 `0`(전체 Pass)이다.
