# Spec 03: 메신저 대화 로그 분석 파이프라인

> **대응 요구사항:** FR-3.1, FR-3.2
> **담당:** 백엔드(조원희) + 비즈니스 로직(김휘중)

---

## FR-3.1 — 카카오톡 내보내기 파일 파싱

### 가치
조장이 별도 가공 없이 카카오톡에서 내보낸 파일을 그대로 넣어 발화량을 팀원별로 집계할 수 있다.

### 구현 기준 (How)

| 담당 | 구현 내용 |
|---|---|
| **백엔드** | 파일을 `UTF-8` → `CP949` 순서로 열고, 카카오톡 정규식 패턴으로 줄 단위 파싱을 수행하여 `List[Dict]` 형태로 반환한다. |
| **프론트엔드** | 메신저 로그 파일(`.txt`) 선택 다이얼로그를 제공하고, 파싱 완료 후 파싱된 발화 수를 상태바에 표시한다. |

#### 세부 사항

**카카오톡 `.txt` 포맷:**
- 날짜 구분선: `r"^\d{4}년 \d{1,2}월 \d{1,2}일"`
- 발화자·메시지 분리: `r"^\[(?P<author>.+?)\] \[(?P<ap>오전|오후) (?P<h>\d{1,2}):(?P<m>\d{2})\] (?P<msg>.*)$"`

**반환 형식:**
```python
List[Dict]  # [{"author": str, "timestamp": str, "message": str}]
```

**인코딩 전략:**
- `UTF-8` 우선 시도
- `UnicodeDecodeError` 시 `CP949(EUC-KR)` 재시도

### 수용 기준 (Acceptance Criteria)

| # | 기준 | 판정 방법 |
|---|---|---|
| AC-1 | 카카오톡 형식 `"2024년 1월 5일\n[Alice] [오후 2:30] 안녕하세요\n[Bob] [오후 2:31] 네 안녕하세요"` → `[{"author": "Alice", "message": "안녕하세요"}, {"author": "Bob", "message": "네 안녕하세요"}]` | 리스트 일치 |
| AC-2 | CP949 인코딩 카카오톡 파일 → 한글 깨짐 없음 | 텍스트 무결성 |

## FR-3.2 — 오염된 줄(Line) 건너뛰기(Skip) 방어적 파싱

### 가치
플랫폼 업데이트로 포맷이 일부 변경되어도 앱이 크래시 없이 가용한 데이터를 최대한 추출한다.

### 구현 기준 (How)

| 담당 | 구현 내용 |
|---|---|
| **백엔드** | `for line in file` 스트리밍 순회 중 각 줄을 `try-except`로 감싸고, 파싱 실패 줄은 `skip_count`를 증가시키며 `continue`로 건너뛴다. |
| **비즈니스 로직** | `skip_count`가 1 이상인 경우 경고 플래그(`has_skipped_lines: True`)를 포함하여 FR-5.3 경고 배너 표시 여부를 결정한다. |

#### 세부 사항
- 각 줄 처리를 `try-except Exception` 블록으로 감싼다.
- 파싱 불일치 또는 예외 줄: `skip_count += 1` 후 `continue`
- 반환값: `{"records": List[Dict], "skipped_lines": int}`

### 수용 기준 (Acceptance Criteria)

| # | 기준 | 판정 방법 |
|---|---|---|
| AC-1 | 정상 10줄 + 오염 3줄 → `records` 길이 `10`, `skipped_lines` `3` | 값 일치 |
| AC-2 | 전체 100줄 오염 → `records` 빈 리스트, `skipped_lines` `100`, 예외 미발생 | 정상 반환 |
| AC-3 | 위 두 경우 모두 함수 정상 반환, 앱 프로세스 미종료 | 프로세스 생존 |
