# Spec 04: 최종 기여도 특성 매트릭스 및 산출 알고리즘

> **대응 요구사항:** FR-4.1, FR-4.2, FR-4.3, FR-4.4
> **담당:** 비즈니스 로직(김휘중) + 프론트엔드(이대한)

---

## FR-4.1 — 지표 Min-Max 정규화

### 가치
단위가 다른 이질적인 지표(라인 수, 글자 수, 발화 횟수)를 0~1 척도로 통합하여 공정하게 비교할 수 있다.

### 구현 기준 (How)

| 담당 | 구현 내용 |
|---|---|
| **비즈니스 로직** | `normalize(values: List[float]) -> List[float]` 함수를 구현한다. `max == min` 예외 처리, `round(..., 4)` 반올림, 결과 범위 보장(`0.0 ≤ v ≤ 1.0`)을 포함한다. |

#### 세부 사항
- 공식: `normalized = (x - min) / (max - min)`
- `max == min`(모든 값 동일): 전원 `0.5` 반환
- 대상 지표 3종: Git 추가 라인 수, OOXML 텍스트 글자 수, 메신저 발화 횟수
- 반올림: `round(..., 4)` (소수점 4자리)

### 수용 기준 (Acceptance Criteria)

| # | 기준 | 판정 방법 |
|---|---|---|
| AC-1 | 입력 `[0, 50, 100]` → `[0.0, 0.5, 1.0]` | 정확히 일치 |
| AC-2 | 입력 `[75, 75, 75]` → `[0.5, 0.5, 0.5]`, `ZeroDivisionError` 미발생 | 값 일치 + 예외 없음 |
| AC-3 | 모든 반환값이 `0.0 ≤ v ≤ 1.0` 범위 내 | 범위 검증 |

## FR-4.2 — 이상치 Capping 및 로그 스케일링

### 가치
대용량 코드 복사·붙여넣기 등 비정상 기여로 인한 점수 왜곡을 방지하여 평가의 신뢰도를 높인다.

### 구현 기준 (How)

| 담당 | 구현 내용 |
|---|---|
| **비즈니스 로직** | 커밋별 추가 라인 수가 1,000을 초과하면 1,000으로 Capping하고, 팀원 합산 후 `math.log1p(x)` 로그 스케일링을 적용하는 전처리 함수를 구현한다. `capping_applied` 플래그를 반환값에 포함한다. |
| **프론트엔드** | Capping이 적용된 경우(`capping_applied: True`) 대시보드에 `"일부 커밋이 이상치 처리되었습니다."` 안내 문구를 표시한다. |

#### 세부 사항
- Capping 임계치: 단일 커밋 추가 라인 수 > `1,000줄` → `1,000`으로 강제 고정
- 로그 스케일: `math.log1p(x)` (밑 e, 0 입력 보호)
- 반환값에 `capping_applied: True/False` 플래그 포함

### 수용 기준 (Acceptance Criteria)

| # | 기준 | 판정 방법 |
|---|---|---|
| AC-1 | 단일 커밋 추가 `5,000` → 내부 집계 값 `1,000` | 값 확인 |
| AC-2 | 추가 `999` → Capping 없이 원본값 `999` 사용 | 값 확인 |
| AC-3 | Capping 적용 시 → `"capping_applied": True` 포함 | 플래그 확인 |

## FR-4.3 — 메신저 파이프라인 결측 시 동적 가중치 재조정

### 가치
메신저 데이터가 없어도 앱이 Git·OOXML 데이터로 기여도를 정상 산출하여 평가를 중단 없이 완료할 수 있다.

### 구현 기준 (How)

| 담당 | 구현 내용 |
|---|---|
| **비즈니스 로직** | `calculate_score(git, doc, msg, weights)` 함수에서 `msg`가 `None`인 경우 동적 가중치 재조정 로직을 구현한다. |
| **백엔드** | 메신저 파이프라인의 성공·실패 여부를 상위 오케스트레이터에 반환하여 결측 여부를 알린다. |

#### 세부 사항
- 기본 가중치: `W_git = 0.4, W_doc = 0.4, W_msg = 0.2`
- 결측 시: `W_msg = 0.0`, 나머지 상대 비율 유지 스케일링
- 공식: `W_i_new = W_i / (1.0 - W_msg)`
- 재조정 결과 예: `W_git_new = 0.5, W_doc_new = 0.5`
- 재조정 후 assert: `round(sum(weights), 4) == 1.0`
- 최종 점수: `Σ (normalized_i × W_i_new)`, 소수점 4자리 반올림

### 수용 기준 (Acceptance Criteria)

| # | 기준 | 판정 방법 |
|---|---|---|
| AC-1 | 메신저 `None` → 예외 없이 기여도 점수 딕셔너리 반환 | 정상 반환 |
| AC-2 | 내부 가중치 합 `1.0` (± 0.0001) | 합계 검증 |
| AC-3 | 메신저 정상 vs 부재 시 → **다른** 점수 반환 | 값 차이 확인 |

## FR-4.4 — 직무별 가중치 프리셋 및 슬라이더 수동 조정

### 가치
조장이 프로젝트 성격에 따라 평가 기준을 직접 설정하여 맞춤형 기여도 평가를 수행할 수 있다.

### 구현 기준 (How)

| 담당 | 구현 내용 |
|---|---|
| **프론트엔드** | 프리셋 버튼 3개, 슬라이더 3개, 가중치 합계 경고 레이블, `[분석 시작]` 버튼 활성화/비활성화 제어 UI를 구현한다. |
| **비즈니스 로직** | `validate_weights(w_git, w_doc, w_msg) -> bool` 함수와 가중치 프리셋 상수 테이블을 구현한다. |

#### 프리셋 테이블

| 프리셋 이름 | W_git | W_doc | W_msg |
|---|---|---|---|
| 개발 중심 | 0.60 | 0.25 | 0.15 |
| 기획 중심 | 0.20 | 0.60 | 0.20 |
| 균형 설정 | 0.40 | 0.40 | 0.20 |

#### 세부 사항
- 슬라이더 범위: `0.00 ~ 1.00`, 스텝 `0.05`
- 합계 != 1.0: `[분석 시작]` 버튼 비활성화 + 경고 텍스트 `"가중치 합계가 1.00이어야 합니다. 현재: X.XX"`

### 수용 기준 (Acceptance Criteria)

| # | 기준 | 판정 방법 |
|---|---|---|
| AC-1 | `[개발 중심]` 클릭 → 슬라이더 `0.60 / 0.25 / 0.15` | 값 확인 |
| AC-2 | `0.5/0.5/0.5` (합 1.5) → 버튼 비활성화 + 경고 표시 | UI 상태 |
| AC-3 | `0.5/0.3/0.2` (합 1.0) → 버튼 활성화 | UI 상태 |
