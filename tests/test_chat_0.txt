* **백엔드/분석 엔진:** Python (로컬 프로세스로 구동, 프론트엔드와 IPC 또는 REST/gRPC 통신)
* **외부 연동 (API):** Google Workspace API (Drive API, Docs API), OAuth 2.0
* **데이터 시각화 및 리포팅:** `matplotlib`, PDF 생성 라이브러리(ReportLab 등)

* 불용어(Stop-words: "넵", "ㅇㅇ", "아하" 등) 사전을 기반으로 무의미한 리액션을 필터링하고, 순수 유효 글자 수 및 자료(파일/링크) 공유 횟수를 카운팅해야 한다.

# QCE (부탁해 꼬마선장) — Python 데스크톱 팀기여도 정량평가앱

## 메타
- 표준: IEEE 830-1998 SRS, 문서 v1.0, 2026-05-27
- 유형: Python 독립 실행형 Standalone 데스크톱앱
- 사용자: 조장 단일사용자 전용 (다중사용자·RBAC 미지원)
- 환경: 조장 PC 1대, 일반 데스크톱/노트북, 특별 HW 요구 無

## 4대 목표
- O-1 통합분석: .pptx/.docx + 메신저로그 + Git 단일 파이프라인
- O-2 어뷰징방어: 자동생성코드·비정상커밋 통계 탐지·경고
- O-3 시각화: 작업지분율·소통량×작업밀도 등 다각도 리포트
- O-4 완전로컬: 외부 AI/클라우드 없이 조장 PC에서 단독 작동

## Out of Scope
- 실시간 협업모니터링·지속 수집
- 외부 LLM/AI API 호출
- 타 팀원 PC 에이전트·플러그인 설치
- 데이터 외부 서버 전송·클라우드 저장

## 용어
- QCE: Quantitative Contribution Evaluator
- MVC: Model-View-Controller
- OOXML: Office Open XML (.docx/.pptx, ISO/IEC 29500)
- NLP: Natural Language Processing
- Z-Score: (x-μ)/σ
- Isolation Forest: 비지도 이상치탐지
- Global Alias Mapping: 다플랫폼 사용자명↔실제팀원 1:1 매핑
- 불용어 사전: 분석 제외 단어/표현 목록
- Air-Gapped: 외부망 단절 독립 운영

## FR-1 신원통합·OOXML파싱 (Identity & OOXML Parsing)
- 1.1 Drag&Drop 업로드 [이대한+조원희] (필수)
  - 입력: .pptx/.docx 1개 이상
  - 처리: 확장자 검증 → 임시경로 복사 → 파싱 모듈 전달
  - 출력: UI 파일목록 + 파싱준비 상태
  - 예외: 미지원 확장자 → 오류 메시지
- 1.2 OOXML 메타데이터 파싱 [조원희] (필수)
  - 입력: 업로드 OOXML
  - 처리: zipfile 해제 → core.xml(dc:creator, cp:lastModifiedBy) +
         app.xml(페이지수·단어수 등) → byte 수정량 산출
  - 출력: 문서별 기여자·수정량 DataFrame
  - 예외: 손상 파일/메타부재 → 경고 + 해당 파일 skip
- 1.3 Global Alias Mapping [이대한+김휘중] (필수)
  - 입력: 각 소스 사용자 식별자 목록
  - 처리: 고유식별자 추출 → 매핑테이블 UI 표시 → 드롭다운/수동 매핑 → 저장
  - 출력: Dict {alias → team_member_entity}
  - 예외: 미매핑 alias → 경고 + 분석 제외 옵션

## FR-2 코드 정량평가 + 조기경보 (Early Warning System)
- 2.1 Git log 정량평가 [조원희] (필수)
  - 입력: 로컬 Git 저장소 경로 또는 git log 텍스트 파일
  - 처리: git log --numstat --format 또는 텍스트 파싱
         → 커밋별 작성자·날짜·추가/삭제행 → 팀원 합산 → AliasMap 적용
  - 출력: 팀원별 커밋수·추가행·삭제행·변경파일수
- 2.2 조기경보 플래그 [김휘중+이대한] (필수)
  - EW-01: 단일커밋 ≥1000줄 → "대량 코드변경 감지(자동생성 의심)"
  - EW-02: 24h내 커밋빈도 ≥ 팀평균×3 → "비정상적인 수의 push 감지"
  - 출력: 대시보드 ⚠️ 경고 메시지
  - 예외: 초기 보일러플레이트 커밋은 수동 예외처리 옵션

## FR-3 메신저 텍스트마이닝 + 포맷어댑터
- 3.1 대화로그 입력·포맷감지 [조원희] (필수)
  - 입력: .txt 또는 .csv
  - 처리: 인코딩 감지(UTF-8/EUC-KR 등) → 헤더 분석 → 플랫폼 식별 → 어댑터 선택
  - 출력: 정규화 스키마 {timestamp, speaker, message}
  - 지원 어댑터
    · 카카오톡 Windows: UTF-8 BOM, "2026년 5월 27일 오전 1:00"
    · 카카오톡 macOS:   UTF-8,     "2026. 5. 27. 오전 1:00:00"
    · 슬랙:             UTF-8 JSON/CSV, ISO8601 "2026-05-27T01:00:00Z"
- 3.2 규칙기반 NLP 필터링 [김휘중] (필수)
  - 입력: 정규화 대화기록
  - 처리: (1) Regex로 timestamp/speaker/message 분리
         (2) KoNLPy 형태소 분석
         (3) 불용어 사전 매칭 필터
         (4) 유효 글자수 산출
  - 출력: 팀원별 유효 소통글자수·유효 메시지수
  - 불용어 카테고리 (UI 추가/수정/삭제 가능)
    · 단순긍정: 넵/네/넹/넴/응/웅/앙
    · 단순리액션: ㅇㅇ/ㅋㅋ/ㅎㅎ/ㄱㄱ/ㄴㄴ
    · 감탄·이해: 아하/아~/오오/헐/와
    · 미디어태그: (이모티콘)/(사진)/(동영상)

## FR-4 통계 분석 + 이상치 탐지 [김휘중]
- 4.1 다차원 특성매트릭스 (필수)
  - 입력: FR-1/2/3 산출 팀원별 지표
  - 처리: Min-Max 또는 Z-Score 정규화 → 팀원(행)×지표(열)
  - 컬럼: 문서수정량(byte)·코드추가행·코드삭제행·커밋수·유효소통글자수·유효메시지수
  - 출력: 정규화 pandas DataFrame
- 4.2 IsolationForest (필수)
  - 입력: 정규화 매트릭스
  - 처리: 모델 훈련(contamination 자동/수동) → 팀원별 anomaly score
  - 출력: 정상/이상치 판정 + score
- 4.3 Z-Score 하위이상치 (필수)
  - 처리: 지표별 μ, σ → Z=(x-μ)/σ → Z<-1.5 지표 ≥2개인 팀원 → 하위이상치
  - 출력: 팀원별·지표별 Z-Score + 판정결과

## FR-5 시각화 리포트 [이대한] (5.4는 김휘중 협업)
- 5.1 작업지분율 파이차트 (필수)
  - PyQt6 캔버스에 pyqtgraph 또는 matplotlib 렌더링
- 5.2 소통량(X)×문서작업밀도(Y) 산점도 (필수)
  - 팀원명 라벨 + 평균선(Mean Line)
- 5.3 고급 시각화 (선택)
  - 타임라인 추이 라인차트
  - 메신저 멘션/응답 빈도 네트워크 그래프
  - 지표별 레이더차트 또는 수평 바차트
  - 대시보드 탭/슬라이드 형태
- 5.4 경보 강조표시 (필수)
  - 경보 팀원 차트요소(파이조각·산점도점)에 붉은색 + ⚠️ 오버레이

### FR-1: OOXML 문서 분석 파이프라인 *(조원희)*

#### FR-1.1 — .pptx / .docx 글자 수 추출
- 구현
  - `python-pptx`, `python-docx`로 XML 파싱
  - .pptx: `prs.slides`의 모든 `shape.text_frame.text` 순회
  - .docx: `document.paragraphs`의 `paragraph.text` 순회
  - 공백·개행 제외 순수 글자 수: `len(text.replace(' ', '').replace('\n', ''))`
  - 도형(shape) 개수 별도 카운트
- 수용 기준
  - 200자 .docx → `200 ± 0`자
  - 슬라이드 3장 × 텍스트박스 2개(각 50자) → 글자 `300`, 도형 `6`

#### FR-1.2 — 팀원 식별자 기반 집계
- 구현
  - 결과 스키마: `Dict[str, int]` = `{"팀원이름": 총_글자수}`
  - 작성자: .docx → `core_properties.author`, .pptx → `core_properties.last_modified_by`
  - 식별자가 `""` 또는 `None`이면 `"Unknown"` 키로 분류
- 수용 기준
  - Alice 100자 + Bob 150자 → `{"Alice":100, "Bob":150}`
  - 메타데이터 없음 → `"Unknown"` 키 존재, 예외 없음

#### FR-2.1 — git log 커밋 데이터 추출
- 구현
  - 호출: `subprocess.run(["git","log","--numstat","--format=%H|%ae|%ai"], cwd=repo_path, capture_output=True, text=True, timeout=30)`
  - 스키마: `{"author_email": {"commits": int, "additions": int, "deletions": int}}`
  - `subprocess.CalledProcessError`·`FileNotFoundError` → 빈 딕셔너리 `{}` 반환, 예외 상위 전파 금지
  - 타임아웃 **30초 고정**
- 수용 기준
  - 커밋 3개(`alice@test.com`, +10/-5) → `{"alice@test.com": {"commits":3, "additions":10, "deletions":5}}`
  - 잘못된 경로(예: `C:\not_a_repo`) → `{}`, 예외 없음
  - 10,000 커밋 저장소 → 30초 이내 완료

#### FR-2.2 — Git 헬스체크 + 안내 팝업
- 구현
  - 메인 윈도우 표시 **전** `subprocess.run(["git","--version"], timeout=5)` 1회 호출
  - `FileNotFoundError` 또는 반환 코드 `!= 0` → 모달 팝업 표시
    - 경고 문구: `"Git이 설치되어 있지 않거나 PATH에 등록되지 않았습니다."`
    - 다운로드 링크: `https://git-scm.com/download/win` (기본 브라우저로 열기)
    - PATH 설정 방법 안내 1줄 이상
    - `[확인]` 버튼 → 팝업 닫고 Git 분석 기능만 비활성화한 채 앱 계속 실행
- 수용 기준
  - Git 미설치 환경 → 메인 윈도우 표시 전 팝업 표시
  - 확인 클릭 후 앱 정상 실행
  - Git 정상 설치 → 팝업 미표시

### FR-3: 메신저 대화 로그 분석 *(조원희)*

#### FR-3.1 — 카카오톡 / 슬랙 파싱
- 구현
  - 카톡 `.txt` 정규식
    - 날짜: `r"^\d{4}년 \d{1,2}월 \d{1,2}일"`
    - 발화: `r"^(.+?)\s:\s(.+)$"`
  - 슬랙 `.csv`: `csv.DictReader`, 필수 컬럼 `user`, `text`, `timestamp`
  - 반환: `[{"author": str, "timestamp": str, "message": str}, ...]`
  - 인코딩: UTF-8 우선 → 실패 시 CP949(EUC-KR) 재시도 (NFR-3.1)
- 수용 기준
  - 카톡 형식 정확 파싱
  - 슬랙 CSV 3행 → 리스트 길이 `3`
  - CP949 한글 깨짐 없음

#### FR-3.2 — 오염 줄 Skip 방어 파싱
- 구현
  - 줄 단위 순회 + `try-except Exception` 블록
  - 예외 발생 시 `skip_count += 1` + `continue`
  - 반환: `{"records": List[Dict], "skipped_lines": int}`
- 수용 기준
  - 정상 10 + 오염 3 → `records` 길이 10, `skipped_lines` 3
  - 전체 100 오염 → `records=[]`, `skipped_lines=100`, 예외 없음
  - 두 경우 모두 정상 return, 프로세스 종료 없음

### FR-4: 최종 기여도 산출 알고리즘 *(김휘중)*

#### FR-4.1 — Min-Max 정규화
- 구현
  - 공식: `normalized = (x - min) / (max - min)`
  - `max == min` (전원 동일) → 전원 `0.5` 반환
  - 대상 지표 3종: Git 추가 라인, OOXML 글자 수, 메신저 발화 횟수
  - 결과는 `round(_, 4)` 적용
- 수용 기준
  - `[0, 50, 100]` → `[0.0, 0.5, 1.0]`
  - `[75, 75, 75]` → `[0.5, 0.5, 0.5]`, `ZeroDivisionError` 없음
  - 모든 결과 `0.0 ≤ v ≤ 1.0`

#### FR-4.2 — 이상치 Capping + 로그 스케일링
- 구현
  - 단일 커밋 추가 라인 > 1,000줄 → `1,000`으로 강제 Capping
  - Capping 이후 합산 → `math.log1p(x)` 적용 → 정규화
  - Capping 적용 커밋 1개 이상 시 반환에 `"capping_applied": True` 포함
- 수용 기준
  - 단일 커밋 5,000 입력 → 내부 집계값 1,000
  - 999 입력 → 999 그대로 사용 (Capping 없음)
  - Capping 발생 시 반환 dict에 `"capping_applied": True`

#### FR-4.3 — 메신저 결측 시 동적 가중치 재조정
- 구현
  - 기본 가중치 예시: `W_git=0.4, W_doc=0.4, W_msg=0.2` (FR-4.4 균형 프리셋)
  - 메신저 데이터 부재 → `W_msg = 0.0`
  - 나머지 재조정: `W_i_new = W_i / (1.0 - W_msg)`
    - 재조정 결과: `W_git_new=0.5, W_doc_new=0.5`
  - 가중치 합 `round(sum, 4) == 1.0` assert로 내부 검증
  - 최종 점수 = `Σ(normalized_i × W_i_new)`, 4자리 반올림
- 수용 기준
  - 메신저 입력 `None`일 때 예외 없이 점수 dict 반환
  - 내부 가중치 합 `1.0 ± 0.0001`
  - 메신저 정상 vs 부재 → **다른** 점수 반환

#### FR-4.4 — 직무별 가중치 프리셋 + 슬라이더
- 프리셋 (UI 버튼 3개)

  | 프리셋 | W_git | W_doc | W_msg |
  |---|---|---|---|
  | 개발 중심 | 0.60 | 0.25 | 0.15 |
  | 기획 중심 | 0.20 | 0.60 | 0.20 |
  | 균형 설정 | 0.40 | 0.40 | 0.20 |
- 슬라이더: 각 지표 `0.00 ~ 1.00`, 스텝 `0.05`
- 합 ≠ 1.0 → `[분석 시작]` 비활성화 + `"가중치 합계가 1.00이어야 합니다. 현재: X.XX"`
- 수용 기준
  - `[개발 중심]` 클릭 → 슬라이더 `0.60 / 0.25 / 0.15`
  - 합 1.5 → 버튼 비활성화 + 경고 표시
  - 합 1.0 → 버튼 활성화

### FR-5: 시각화 및 대시보드 *(이대한)*

#### FR-5.1 — 팀원별 기여도 차트
- 구현
  - `matplotlib`(또는 동등) 사용
  - 종합 막대 차트: X=팀원명, Y=최종 점수 (0.0~1.0)
  - 지표별 레이더(방사형) 차트: Git · 문서 · 메신저 3 꼭짓점
  - 분석 완료 시 자동 표시
  - 분석 미실행 패널 텍스트: `"분석을 실행하면 결과가 표시됩니다."`
- 수용 기준
  - 팀원 4명 → 막대 4개 + 팀원명 레이블
  - 레이더 꼭짓점 레이블 `"Git" / "문서" / "메신저"` 3개 표시

#### FR-5.2 — 리포트 파일 저장 (.md / .csv)

  - `.md` 텍스트 에디터에서 테이블 정상 렌더링
  - `.csv` Excel에서 한글 깨짐 없음
  - 상태바 텍스트 3초 표시 후 사라짐

#### FR-5.3 — 결측 데이터 경고 인터페이스

## 5. 비기능 요구사항 (NFR)

### NFR-1: UI 비동기 처리 *(이대한)*
- 구현
  - 파싱·분석 전체를 메인 스레드와 분리된 **Worker Thread** (`QThread+QRunnable`, `threading.Thread` 등)에서 실행
  - 시작 시 진행률 표시줄(Progress Bar) 표시
  - 완료/오류 시 Signal/Callback으로 메인 스레드에 통보
- 수용 기준
  - 10만 줄 분석 중 메인 윈도우 드래그 가능 (프리징 없음)
  - 시작 1초 이내 진행률 표시
  - 완료·오류 시 진행률 사라짐 + 결과/메시지 표시

### NFR-1.2: 분석 중복 실행 방지 (Race Condition Guard)
- 구현
  - 전역 플래그 `is_analyzing: bool = False`
  - `[분석 시작]` 클릭 시 `True`면 즉시 `return`
  - Worker 시작 직전: `is_analyzing=True` + 버튼 `setEnabled(False)`
  - Worker 종료 시 (성공·오류·취소 무관): `is_analyzing=False` + 버튼 `setEnabled(True)`
  - 상태 복원은 메인 스레드 Signal/Callback에서만 (Worker 내부 직접 UI 수정 금지)
- 수용 기준
  - 시작 후 **0.5초** 이내 버튼 비활성화
  - 5회 연속 클릭 → Worker 1개만 실행 (`threading.active_count()` 검증)
  - 재분석 결과가 첫 실행과 동일 (중복 누적 없음)
  - `RuntimeError`로 비정상 종료 시에도 버튼 재활성화

### NFR-2: 데이터 보안 및 프라이버시

#### NFR-2.1 — 읽기 전용 파일 접근
- 분석 대상은 `'r'` / `'rb'` 모드만 사용. `'w'/'a'/'x'` 절대 금지
- 리포트 저장은 사용자가 지정한 새 경로에만 쓰기
- 수용 기준
  - 입력 원본 파일 `os.path.getmtime`이 분석 전후 동일
  - 코드 리뷰 시 분석 대상에 `open(path,'w')` 또는 `shutil.move/delete` 발견 → Fail

#### NFR-2.2 — 네트워크 통신 완전 차단
- `requests`, `urllib`, `httpx`, `socket`, `http.client` 임포트 금지
- 링크 열기는 `webbrowser.open()` (OS 위임)만, 앱 내부 HTTP 요청 생성 금지
- 수용 기준
  - 방화벽·Wireshark로 외부 IP 송신 패킷 **0건**
  - 코드에 네트워크 라이브러리 import 발견 → Fail

#### NFR-2.3 — 익명화 캐시 (원자적 쓰기)
- 직렬화: `json.dumps()`**만** 사용. `pickle` 모듈 import 전면 금지
- 캐시 저장 **허용** 데이터: 팀원 식별자(이름·이메일), 정규화 점수, 분석 타임스탬프, 가중치 설정값
- 캐시 저장 **금지** 데이터: 메시지 본문(`str`), 소스코드 파일 내용
- 원자적 쓰기 3단계
  1. 동일 디렉터리 임시 파일 `.qce_cache.tmp`에 JSON 완전 쓰기
  2. `tmp_file.flush()` + `os.fsync(tmp_file.fileno())`로 디스크 기록 보장
  3. `os.replace('.qce_cache.tmp', '.qce_cache')` 단일 원자 교체
- 손상 캐시 복구
  1. 로드 중 `json.JSONDecodeError` 또는 `KeyError` → `.qce_cache` 즉시 `os.remove`
  2. 모달: `"캐시 파일이 손상되어 삭제되었습니다. 재분석이 필요합니다."`
  3. 빈 초기 상태로 앱 정상 시작
- 수용 기준
  - 캐시 `json.loads()` 시 점수 dict 정상 복원
  - 캐시 내에 메신저 원본 메시지 문자열 존재 → Fail
  - 코드에 `import pickle` 또는 `from pickle` 발견 → Fail
  - 쓰기 중 `taskkill /F` 후 재시작 → 다이얼로그 표시 + 초기 화면
  - 정상 쓰기 후 `.qce_cache.tmp` 잔존 → Fail

#### NFR-2.4 — 원본 파일 없이 캐시만으로 로드
- `.qce_cache` 존재 시 원본 파일 검사 없이 즉시 역직렬화 → 대시보드 표시
- 상태바: `"캐시 파일에서 이전 분석 결과를 불러왔습니다. (분석 일시: YYYY-MM-DD HH:MM)"`
- 수용 기준
  - 원본 `.txt`, `.docx`, Git 저장소 폴더 전체 삭제 후 재시작 → 차트·점수 정상 표시
  - 상태바 캐시 로드 메시지 표시

### NFR-3: 안정성·테스트 가능성

#### NFR-3.1 — 인코딩 자동 감지
- 시도 순서
  1. `open(path,'r',encoding='utf-8')`
  2. `UnicodeDecodeError` 시 `encoding='cp949'` 재시도
  3. 둘 다 실패 → 파일 skip + `{"error":"encoding_failed","path":path}` 반환
- 3단계 후 앱 계속 실행
- 수용 기준
  - UTF-8 → 1회 성공
  - CP949 → UTF-8 실패 후 CP949로 성공, 한글 정상
  - Shift-JIS 등 → 종료 없이 `error` 키 dict 반환

#### NFR-3.2 — 메신저 모듈 결합도 완화 (격리 테스트)
- `messenger_parser`를 독립 모듈로 분리
- `git_parser`, `ooxml_parser`는 `messenger_parser`를 직접 import 금지
- 공통 인터페이스: `Dict[str, int]`
- 통합은 상위 오케스트레이터(`analyzer.py` 등)에서 수행
- 수용 기준
  - `messenger_parser.py` 삭제·파일명 변경 후 `git_parser`/`ooxml_parser` 단위 테스트 (`pytest`) `ImportError` 없이 통과
  - `messenger_parser`가 `RuntimeError` Mock → Git·OOXML 결과는 정상값 포함
  - `pytest` 종료 코드 `0`

## FR-4 최종 산출 [김휘중]
- 4.1 Min-Max 정규화

  - 수용기준
    · 단일 5000 → 내부집계 1000
    · 999 → 999 그대로
    · Capping 시 플래그 True
- 4.3 메신저 결측 → 동적 가중치 재조정
  - 기본 W_git=0.4, W_doc=0.4, W_msg=0.2 (균형 프리셋)
  - 메신저 부재 → W_msg=0, W_i_new = W_i/(1.0-W_msg)
    · 결과 W_git_new=0.5, W_doc_new=0.5

  - 최종 = Σ(normalized_i × W_i_new), 4자리 반올림
  - 수용기준
    · 메신저 None → 예외X 점수반환

    · 메신저 유/무 결과 다름

[조원희] [3:38] 1. NFR-2.1: 남의 원본 파일 건드리면 죽는다 (읽기 전용)
핵심: 분석할 원본 파일은 건드리지 말고 눈으로만 봐라.

풀이: 파일 열어서 분석한답시고 원본 파일 수정 일자(타임스탬프) 바뀌거나 내용 건드리면 바로 빠꾸(Fail) 먹임. 결과물 리포트 저장할 거면 엉뚱한 데 덮어쓰지 말고, 유저가 "여기다 저장해"라고 지정한 새 폴더에만 얌전히 저장해라.

2. NFR-2.2: 인터넷 선 뽑고 일해라 (네트워크 차단)
핵심: 프로그램에서 몰래 외부로 데이터 빼돌리는 짓(백도어 등) 절대 금지.

풀이: 분석하다가 몰래 외부 IP로 뭐 전송하는 거 걸리면 뒤짐. 나중에 코드 까봤을 때 네트워크 연결하는 라이브러리 한 줄이라도 있으면 바로 컷(Fail). 유저가 외부 링크 클릭하면 니들 프로그램 안에서 띄우지 말고 그냥 윈도우 기본 브라우저(크롬 등)한테 토스해라.

3. NFR-2.3: 찌꺼기 파일에 민감한 내용 남기지 마라
핵심: 임시 저장(캐시)할 때 남의 톡방 본문이나 소스 코드 원본 절대 저장하지 마라.

풀이: * 캐시 파일 털렸을 때 내용물 들통나면 안 되니까 점수, 타임스탬프 같은 껍데기만 JSON으로 예쁘게 저장해라.

파이썬 쓰면 보안 개나 줘버린 pickle 라이브러리 쓰지 마라. 발견되면 컷.

만약 저장하다가 강제 종료돼서 캐시 파일 뻑났다? 복구한답시고 똥꼬쇼 하지 말고 쿨하게 파일 삭제해 버려. 그다음 유저한테 "파일 뻑나서 지웠음ㅇㅇ 첨부터 다시 하셈" 알림창 띄워라.

다 썼으면 디스크에 쓰레기(임시 파일) 남기지 말고 깔끔하게 치워라.

4. NFR-2.4: 원본 없어도 화면은 띄워라 (캐시 단독 로드)
핵심: 유저가 원본 파일 삭제했어도 예전 분석 결과는 보여줘라.

풀이: 원본 파일 날아갔다고 프로그램 먹통 만들지 마라. 남아있는 캐시 파일만 긁어모아도 예전에 뽑아둔 차트나 점수는 띄워줘야 됨. 대신 화면 구석 상태바에 "이거 원본 없고 예전 캐시 파일에서 불러온 거임 (분석 날짜)"라고 핑계 댈 문구는 꼭 적어놔라.

Claude Code(로컬 설치형)를 사용하면 해결된다는 것을 확인 → 실제로 설치 후 실행


역할 분담 방향

기존의 Backend / Business Logic / Frontend 역할 분담은 순서 충돌 문제가 있음

예: Frontend 담당자는 Backend가 완성될 때까지 아무것도 못 함


대신 STAGE 순서 기반으로 역할을 나누는 것이 더 효율적

  김휘중 → STAGE 1, 4, 5  (수치 계산·분석 로직)

  이대한 → STAGE 7        (UI·차트, STAGE 4 완료 후 시작)

단, 김휘중의 STAGE 1이 먼저 완료되어야 나머지 두 명이 시작 가능

제일 중요한 결정입니다. 배경부터요.
차트를 그리려면 팀원별 점수 묶음이 필요해요. 그게 MemberScore라는 데이터 덩어리입니다 (이름, git 점수, 문서 점수, 종합 점수 같은 걸 담은 한 사람치 카드라고 보면 됩니다). 그런데 이 MemberScore는 원래 Model(계산하는 쪽)이 만들어내는 물건이에요.
문제는 이 프로젝트의 절대 규칙(C-4)이 **"View는 Model을 절대 import하면 안 된다"**는 거예요. 정적 분석으로 "View가 Model을 가져다 쓴 횟수 = 0"이어야 통과입니다. 그런데 차트(View)가 MemberScore를 쓰려고 from model import MemberScore 하는 순간 이 규칙을 어기게 됩니다.
그래서 "그럼 MemberScore를 누구 소유로 둘 거냐"를 정해야 합니다. 두 가지 길이 있어요.

제 제안: MemberScore를 Model도 View도 아닌 중립 지대(common/dto.py 같은 공용 폴더)에 둡니다. 그러면 Model도 거기서 꺼내 쓰고 View도 거기서 꺼내 쓰니까, "View가 Model을 import한" 게 아니게 됩니다. 규칙 위반이 아니에요. 양쪽이 공용 창고를 쓰는 셈.
더 빡센 대안: 차트한테 MemberScore 같은 깔끔한 객체를 아예 안 주고, Controller가 {"이름": "홍길동", "git": 0.5, ...} 같은 **그냥 딕셔너리(dict)**로 풀어서 넘깁니다. 차트는 점수 객체가 뭔지조차 모르고 그냥 숫자만 받아요. 더 안전하지만 코드가 좀 지저분해지고(score.git_score 대신 data["git"]), 점수 항목이 9개라 손이 더 갑니다.

Context (왜 / 무엇을)
SW공학 팀프로젝트 채점 기준으로 운영 소스(qce/ + main.py) 5,000줄이 필요하다.

제약: 가식적 줄바꿈/포맷팅, 대량 Docstring·주석 추가는 금지. 실제 동작하는 코드로만 채운다.
집계 제외: tests/·BusinessLogic/(테스트, 이미 ~3,870줄), src/(고아 코드).
사전 조사로 확정된 사실 (계획 전제)
src/는 완전한 고아 코드다. main.py·view_demo.py·QCE.spec·전체 테스트가 qce.*만 import한다. src를 import하는 건 src/ 내부 4개 파일뿐(src 밖에서 참조 0건). PyInstaller도 collect_all('qce')만 번들링. → src/ 삭제해도 앱·빌드·테스트 무영향. (메모리 parser-design-decisions.md의 D-3 "정식경로=src/"는 현재 사실과 반대 → 정정 필요.)
