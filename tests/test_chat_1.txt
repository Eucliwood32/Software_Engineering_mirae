소프트웨어 요구사항 명세서 (SRS) - 코드싸개 식별기

1. 시스템 개요 본 시스템은 협업 프로젝트 내 잉여 인력의 위장 기여를 데이터 기반으로 낱낱이 파헤치는 로컬 구동형 정적 분석 도구(.exe)이다. 복잡한 외부 API나 구글 문서 파싱 같은 위험한 짓은 덜어내고, 오직 안전하고 합법적인 Git 로그 분석과 AST 파싱에만 집중하여 팀원들의 멘탈을 붕괴시키는 것을 목표로 한다.

2. 기능적 요구사항 (Functional Requirements)

FR-01: 다국어 AST 기반 기여도 심판기 (The Multilingual Judge)

사용자가 던져준 로컬 Git 저장소를 스캔하여, C++ 파서와 Kotlin 파서를 각각 호출해 추상 구문 트리를 분석한다.

실질적 로직(Control Flow) 변경 없이 변수명만 끄적거리거나 띄어쓰기만 추가한 얌체 커밋을 식별해 '실질 기여율 0%'로 환산한다.

FR-02: 커밋 메시지 진실 번역기 (Bullshit-to-Truth Converter)

성의 없는 "Update"나 "Fix bug" 같은 커밋 메시지를 로컬 정규식을 통해 "사실 아무것도 안 했지만 커밋 수는 채워야겠음"으로 번역하여 최종 리포트에 기록한다.

FR-03: 올빼미족 색출 히트맵 (Procrastination Heatmap)

마감일 직전 24시간에만 커밋이 몰려있는 패턴을 시각화하여, "이 조원의 코드는 벼락치기와 복붙의 산물임"을 증명하는 붉은색 히트맵을 생성한다.

FR-04: 소유권 방어 퀴즈 (Ownership Litmus Test)

특정 조원이 남의 코드를 수정하려 할 때, 해당 코드의 시간 복잡도나 구조를 묻는 팝업 퀴즈를 띄운다. 통과하지 못하면 "본인이 짠 코드가 아니므로 접근 금지" 처리한다.

FR-05: 핑계 빙고판 및 반성문 자동화 (The Final Humiliation)

단톡방이나 커밋 로그의 상습 핑계 키워드를 수집해 조원별 빙고판을 만들고, 기여도 최하위 조원의 바탕화면에는 "저는 프로젝트의 기생충입니다"로 시작하는 .docx 반성문 서식을 친절하게 자동 생성해 준다.

3. 비기능적 요구사항 (Non-Functional Requirements)

NFR-01: 완전한 오프라인 구동 (API 무용지물)

외부 데이터 연동은 철저히 배제하며, 모든 분석은 사용자의 로컬 CPU 리소스만을 쥐어짜서 1초 이내에 수행되어야 한다.

NFR-02: 환경 변수 면역력 (Bulletproof Environment)

분석 대상 프로젝트의 특이한 에디터 환경 변수나 개발 환경 설정에 영향받지 않고, 오직 .git 디렉토리의 원시 데이터만 추출해 분석의 무결성을 유지한다

## 🔍 2. 조장 PC 중심의 사후 데이터 마이닝 (Data Mining)

* **OOXML 메타데이터 역추적 (로컬 파일 분석):**

* OS 계정명 및 기기 고유 ID를 기준으로 **"누가 이 도형/텍스트를 최초 생성하고 마지막으로 수정했는지"** 바이트(Byte) 단위 추적


* **클라우드 공유 문서 웹 스크래핑 (웹 로그 분석):**
* 구글 독스, 노션, 캔바 등의 작업 링크를 조장의 데스크톱 앱에 입력

* **커뮤니케이션 텍스트 마이닝 (소통 로그 분석):**
* 단톡방 대화록(`.txt`)을 추출하여 팀원별 발언 빈도 및 자료 공유 횟수 계산
* 정규표현식(Regex)을 이용해 "넵", "ㅋ", "아하" 같은 **무의미한 단순 리액션 단어는 가중치에서 자동 필터링**

* **수학적 이상치 탐지 (Isolation Forest / $Z$-Score):**
* 열심히 참여한 조원들의 평균값에서 통계적으로 지나치게 멀리 떨어진 음수(-) 방향의 이상치(Outlier)를 연산
* 다른 팀원들이 일할 때 데이터 흔적이 0에 수렴하는 조원을 무임승차 위험군으로 최종 판정

## 📝 프로젝트 요구사항 정의서 (PRD)

**프로젝트명:** 데이터 기반 기여도 증빙 및 무임승차 탐지 솔루션 (가칭: Free-Rider Hunter)
**목표:** 조장(Team Leader)의 데스크톱 PC 단일 환경에서 로컬 파일, 메신저 로그, 클라우드 문서 API를 통합 분석하여 객관적인 팀원 기여도(%) PDF 보고서를 자동 생성한다.

### 1. 시스템 아키텍처 및 기술 스택

* **프론트엔드 (Desktop GUI):** Kotlin 또는 C++ 기반 데스크톱 클라이언트 프레임워크
* **백엔드/분석 엔진:** Python (로컬 프로세스로 구동, 프론트엔드와 IPC 또는 REST/gRPC 통신)
* **외부 연동 (API):** Google Workspace API (Drive API, Docs API), OAuth 2.0
* **데이터 시각화 및 리포팅:** `matplotlib`, PDF 생성 라이브러리(ReportLab 등)

### 2. 기능적 요구사항 (Functional Requirements)

**FR-1: 로컬 문서 메타데이터 추출 (OOXML Parsing)**

* 사용자는 앱 UI에 `.pptx`, `.docx` 파일을 Drag & Drop으로 업로드할 수 있어야 한다.
* 시스템은 파일의 압축을 해제하고 내부 `core.xml`, `app.xml`을 파싱하여, 도형/텍스트 블록별 '최초 생성자'와 '최종 수정자'의 OS 계정명 및 바이트(Byte) 수정량을 추출해야 한다.
* 추출된 OS 계정명을 실제 팀원 이름과 매핑(Mapping)할 수 있는 UI를 제공해야 한다.

**FR-2: 클라우드 문서 API 연동 (Google Docs/Notion)**

* 시스템은 Google OAuth 2.0 인증 인터페이스를 제공하여, 사용자의 구글 계정(읽기 전용 권한) 접근을 승인받아야 한다.
* 사용자가 구글 독스 문서의 URL을 입력하면, 시스템은 `Google Drive Revisions API`를 호출해야 한다.
* API 응답(JSON)을 파싱하여 각 리비전(수정본)의 시간, 작성자 ID, 추가/삭제된 글자 수 데이터를 시계열로 누적해야 한다.

**FR-3: 커뮤니케이션 텍스트 마이닝 (Chat Log Analysis)**

* 사용자는 카카오톡 대화록 내보내기 파일(`.txt`)을 업로드할 수 있어야 한다.
* 시스템은 정규표현식(Regex)을 사용하여 타임스탬프와 발화자를 분리해야 한다.
* 불용어(Stop-words: "넵", "ㅇㅇ", "아하" 등) 사전을 기반으로 무의미한 리액션을 필터링하고, 순수 유효 글자 수 및 자료(파일/링크) 공유 횟수를 카운팅해야 한다.

**FR-4: 기여도 통계 및 이상치 탐지 (Data Analytics)**

* 수집된 3가지 지표(문서 수정 바이트, 클라우드 기여도, 유효 소통 글자 수)를 정규화하여 3차원 특성 매트릭스로 변환해야 한다.
* Isolation Forest 알고리즘을 적용하여 정상 범주에서 벗어난 데이터 포인트를 식별해야 한다.
* 각 지표별 통계적 평균에서 음수(-) 방향으로 이탈한 정도를 `$Z$-Score`로 계산하여 최종 무임승차 위험군을 판정해야 한다.

**FR-5: 최종 보고서 생성 (Reporting & Export)**

* 계산된 팀원별 최종 작업 지분율(%)을 파이 차트로 렌더링해야 한다.
* '소통량(X축)' vs '문서 작업량(Y축)'을 산점도(Scatter Plot)로 시각화하여 작업 밸런스를 증명해야 한다.
* 교수 제출용 A4 포맷의 PDF 보고서로 최종 Export 하는 기능을 제공해야 한다.

### 3. 비기능적 요구사항 (Non-Functional Requirements)

**NFR-1: 제약사항 및 설치 최소화**

* 타 팀원들의 PC에는 어떠한 소프트웨어나 익스텐션도 설치되어서는 안 된다.
* 오직 조장의 데스크톱 1대에서 모든 로직(OAuth 인증 포함)이 실행되어야 한다.

**NFR-2: 보안 및 프라이버시 (Security & Privacy)**

* Google API 인증 시 요구하는 권한(Scope)은 최소한의 '읽기 전용(Read-only)'으로 제한해야 한다.
* 추출된 카카오톡 대화 내용 및 OAuth Token은 외부 서버로 전송되지 않고 철저히 로컬 환경에서만 휘발성으로 처리되어야 한다.

FR-2: 코드/작업량 정량 평가 및 조기 경보 시스템 (Early Warning System)

시스템은 Git 로그 및 파일 작업량을 기반으로 점수를 정량적으로 합산하여 계산한다.

단, 특정 팀원이 단기간 내에 비정상적인 빈도로 Push를 하거나, 단일 커밋/작업의 용량이 임계점(예: 1,000줄 이상)을 초과할 경우, 대시보드 UI 상에 ⚠️ "비정상적인 수의 push 감지 (자동 생성 코드 의심)" 형태의 경고 플래그를 출력하여 조장이 최종 판단을 내릴 수 있도록 보조한다.

FR-3: 커뮤니케이션 텍스트 마이닝 및 포맷 어댑터 (Chat Log Analysis & Adapter)

메신저 대화록 내보내기 파일(.txt, .csv)을 입력받는다.

다중 포맷 어댑터: 파일 헤더를 분석하여 카카오톡(Windows/Mac) 및 슬랙 등 다양한 플랫폼과 OS의 인코딩/시간 포맷을 자동 감지하고 단일 규격으로 정규화한다.

규칙 기반 NLP 필터링: 정규표현식(Regex)으로 타임스탬프와 발화자를 분리하고, KoNLPy와 자체 구축한 '불용어 사전(Stop-words: "넵", "ㅇㅇ", "아하" 등)'을 결합하여 무의미한 단순 리액션을 물리적으로 필터링한 후 순수 유효 글자 수를 산출한다.

FR-4: 기여도 통계 및 이상치 탐지 (Data Analytics)

수집된 지표(문서/코드 작업 밀도, 유효 소통 글자 수)를 pandas를 통해 정규화하여 다차원 특성 매트릭스로 변환한다.

scikit-learn의 Isolation Forest 알고리즘을 로컬 연산으로 가동하여 그룹 내 정상적인 작업 패턴과 동떨어진 개체를 식별한다.

각 지표별 통계적 평균 기여도에서 이탈한 정도를 Z-Score로 계산하여, '기여도 하위 이상치'를 판정한다.

FR-5: 최종 정량평가 생성 (Reporting)

pyqtgraph 또는 matplotlib 모듈을 PyQt6 캔버스에 연동하여 시각화 리포트를 생성한다.

팀원별 실제 작업 지분율(%) 파이 차트, '소통량(X축)' vs '문서 작업 밀도(Y축)' 산점도(Scatter Plot)를 렌더링한다.

FR-2에서 발생한 조기 경보 플래그가 발생한 팀원의 리포트에는 붉은색 시각적 강조 효과를 부여한다.

3. 비기능적 요구사항 (Non-Functional Requirements)
NFR-1: 작동 제약 및 완전한 로컬 독립성 (Air-Gapped Operation)

No AI / No Cloud: 데이터를 분석하는 과정에서 일체의 외부 LLM API(OpenAI, Claude 등)나 클라우드 연산 서버를 호출하지 않는다. 모든 분석은 로컬 PC의 CPU/RAM 자원만을 활용한다.

데이터 수집을 위해 평가 대상인 타 팀원들의 PC에 어떠한 소프트웨어나 익스텐션도 설치를 요구하지 않는다. 오직 조장의 데스크톱 1대에서 모든 병합 및 통계 처리가 이루어진다.

NFR-2: 데이터 보안 및 프라이버시 (Security & Privacy)

외부 API 인증(Git 등) 시 요구하는 권한은 철저히 읽기 전용(Read-only)으로 제한한다.

수집된 대화 내용, 코드 로그, OAuth Token 등은 외부망으로 단 1바이트도 전송되지 않으며, 프로그램 종료 시 로컬 캐시 메모리에서 즉각적으로 휘발 파기된다.

# QCE (부탁해 꼬마선장) — Python 데스크톱 팀기여도 정량평가앱

## 메타
- 표준: IEEE 830-1998 SRS, 문서 v1.0, 2026-05-27
- 유형: Python 독립 실행형 Standalone 데스크톱앱
- 사용자: 조장 단일사용자 전용 (다중사용자·RBAC 미지원)
- 환경: 조장 PC 1대, 일반 데스크톱/노트북, 특별 HW 요구 無

## 4대 목표
- O-1 통합분석: .pptx/.docx + 메신저로그 + Git 단일 파이프라인
- O-2 어뷰징방어: 자동생성코드·비정상커밋 통계 탐지·경고
- O-3 시각화: 작업지분율·소통량×작업밀도 등 다각도 리포트
- O-4 완전로컬: 외부 AI/클라우드 없이 조장 PC에서 단독 작동

## Out of Scope
- 실시간 협업모니터링·지속 수집
- 외부 LLM/AI API 호출
- 타 팀원 PC 에이전트·플러그인 설치
- 데이터 외부 서버 전송·클라우드 저장

## 용어
- QCE: Quantitative Contribution Evaluator
- MVC: Model-View-Controller
- OOXML: Office Open XML (.docx/.pptx, ISO/IEC 29500)
- NLP: Natural Language Processing
- Z-Score: (x-μ)/σ
- Isolation Forest: 비지도 이상치탐지
- Global Alias Mapping: 다플랫폼 사용자명↔실제팀원 1:1 매핑
- 불용어 사전: 분석 제외 단어/표현 목록
- Air-Gapped: 외부망 단절 독립 운영

## 팀 (학번/이름/역할/담당)
- 20222047 조원희 Backend: 데이터파싱엔진·Git분석·채팅어댑터·보안모듈
- 20247142 이대한 Frontend: PyQt6 UI/UX·대시보드·Drag&Drop·시각화차트
- 20221985 김휘중 BusinessLogic: NLP필터링·기여도산출·신원매핑·통계연산
- Controller 공동, SRS 변경은 전원 PR 합의

## 입력 인터페이스
- 로컬파일: .pptx/.docx (OOXML)
- Git: 로컬 저장소 또는 git log 출력 텍스트
- 메신저: 대화 내보내기 .txt/.csv
- SW 인터페이스
  - Git CLI: git log 명령 + 로컬 .git 디렉토리
  - OS 파일시스템: Python os/pathlib
  - OOXML 내부 XML: python-docx/python-pptx/zipfile (core.xml, app.xml)
- HW 인터페이스: 없음

## 출력
- PyQt6 캔버스 시각화 리포트 (대시보드 탭/슬라이드)
- ⚠️ 경고 플래그 UI 표시

## 아키텍처: MVC 엄격 준수
- 의존성 단방향: View → Controller → Model
- View는 Model import 금지, Controller만 Model 호출

## 모듈
- View [이대한]
  - MainWindow — Drag&Drop, 팀원목록, 경보영역 (FR-1.1)
  - AliasMappingView/Dialog — 신원매핑 (FR-1.3)
  - DashboardView — 시각화 (FR-5.1~5.4)
  - WarningPanel — 경보 UI (FR-2.2)
- Controller [공동]
  - AppController — 이벤트수신, V↔M 중재, 데이터 변환/전달
- Model
  - [조원희] DocumentParser(FR-1.2), GitAnalyzer(FR-2.1), ChatAdapter(FR-3.1)
  - [김휘중] AliasMapper(FR-1.3), EarlyWarningEngine(FR-2.2),
            NLPFilter(FR-3.2), ContributionEngine(FR-4.1~4.3)

## 기술 스택
- View: PyQt6|PySide6 6.x, pyqtgraph|matplotlib
- Parse: pandas 2.x, numpy 1.x, python-docx, python-pptx, zipfile
- NLP: KoNLPy(Okt/Komoran) + JDK, re, 자체 불용어사전
- Stats: scikit-learn 1.x (IsolationForest, Z-Score)
- Test: pytest (Model 우선 단위테스트)
- Build: PyInstaller 또는 Nuitka
- 금지: OpenAI/Claude/Gemini 등 외부 LLM, 클라우드 연산서버

## FR-1 신원통합·OOXML파싱 (Identity & OOXML Parsing)
- 1.1 Drag&Drop 업로드 [이대한+조원희] (필수)
  - 입력: .pptx/.docx 1개 이상
  - 처리: 확장자 검증 → 임시경로 복사 → 파싱 모듈 전달
  - 출력: UI 파일목록 + 파싱준비 상태
  - 예외: 미지원 확장자 → 오류 메시지
- 1.2 OOXML 메타데이터 파싱 [조원희] (필수)
  - 입력: 업로드 OOXML
  - 처리: zipfile 해제 → core.xml(dc:creator, cp:lastModifiedBy) +
         app.xml(페이지수·단어수 등) → byte 수정량 산출
  - 출력: 문서별 기여자·수정량 DataFrame
  - 예외: 손상 파일/메타부재 → 경고 + 해당 파일 skip
- 1.3 Global Alias Mapping [이대한+김휘중] (필수)
  - 입력: 각 소스 사용자 식별자 목록
  - 처리: 고유식별자 추출 → 매핑테이블 UI 표시 → 드롭다운/수동 매핑 → 저장
  - 출력: Dict {alias → team_member_entity}
  - 예외: 미매핑 alias → 경고 + 분석 제외 옵션

## FR-2 코드 정량평가 + 조기경보 (Early Warning System)
- 2.1 Git log 정량평가 [조원희] (필수)
  - 입력: 로컬 Git 저장소 경로 또는 git log 텍스트 파일
  - 처리: git log --numstat --format 또는 텍스트 파싱
         → 커밋별 작성자·날짜·추가/삭제행 → 팀원 합산 → AliasMap 적용
  - 출력: 팀원별 커밋수·추가행·삭제행·변경파일수
- 2.2 조기경보 플래그 [김휘중+이대한] (필수)
  - EW-01: 단일커밋 ≥1000줄 → "대량 코드변경 감지(자동생성 의심)"
  - EW-02: 24h내 커밋빈도 ≥ 팀평균×3 → "비정상적인 수의 push 감지"
  - 출력: 대시보드 ⚠️ 경고 메시지
  - 예외: 초기 보일러플레이트 커밋은 수동 예외처리 옵션

## FR-3 메신저 텍스트마이닝 + 포맷어댑터
- 3.1 대화로그 입력·포맷감지 [조원희] (필수)
  - 입력: .txt 또는 .csv
  - 처리: 인코딩 감지(UTF-8/EUC-KR 등) → 헤더 분석 → 플랫폼 식별 → 어댑터 선택
  - 출력: 정규화 스키마 {timestamp, speaker, message}
  - 지원 어댑터
    · 카카오톡 Windows: UTF-8 BOM, "2026년 5월 27일 오전 1:00"
    · 카카오톡 macOS:   UTF-8,     "2026. 5. 27. 오전 1:00:00"
    · 슬랙:             UTF-8 JSON/CSV, ISO8601 "2026-05-27T01:00:00Z"
- 3.2 규칙기반 NLP 필터링 [김휘중] (필수)
  - 입력: 정규화 대화기록
  - 처리: (1) Regex로 timestamp/speaker/message 분리
         (2) KoNLPy 형태소 분석
         (3) 불용어 사전 매칭 필터
         (4) 유효 글자수 산출
  - 출력: 팀원별 유효 소통글자수·유효 메시지수
  - 불용어 카테고리 (UI 추가/수정/삭제 가능)
    · 단순긍정: 넵/네/넹/넴/응/웅/앙
    · 단순리액션: ㅇㅇ/ㅋㅋ/ㅎㅎ/ㄱㄱ/ㄴㄴ
    · 감탄·이해: 아하/아~/오오/헐/와
    · 미디어태그: (이모티콘)/(사진)/(동영상)

## FR-4 통계 분석 + 이상치 탐지 [김휘중]
- 4.1 다차원 특성매트릭스 (필수)
  - 입력: FR-1/2/3 산출 팀원별 지표
  - 처리: Min-Max 또는 Z-Score 정규화 → 팀원(행)×지표(열)
  - 컬럼: 문서수정량(byte)·코드추가행·코드삭제행·커밋수·유효소통글자수·유효메시지수
  - 출력: 정규화 pandas DataFrame
- 4.2 IsolationForest (필수)
  - 입력: 정규화 매트릭스
  - 처리: 모델 훈련(contamination 자동/수동) → 팀원별 anomaly score
  - 출력: 정상/이상치 판정 + score
- 4.3 Z-Score 하위이상치 (필수)
  - 처리: 지표별 μ, σ → Z=(x-μ)/σ → Z<-1.5 지표 ≥2개인 팀원 → 하위이상치
  - 출력: 팀원별·지표별 Z-Score + 판정결과

#   충돌 주제   v2.0/v2.1   새 명세서   중요도
①   NLP 라이브러리   JRE 금지, kiwipiepy만 허용   KoNLPy + JDK 필수   🔴 치명
②   이상치 처리   Capping(1000줄) + log scale   IsolationForest + Z-Score   🔴 치명
③   캐시 정책   .qce_cache 파일 영구 유지   종료 시 전체 휘발 (N2.3)   🔴 치명
④   시각화 차트   막대차트 + 레이더차트   파이차트 + 산점도 + 타임라인   🟡 중요
⑤   문서 기여 단위   순수 글자 수   byte 수정량   🟡 중요
⑥   저장소 구조   specs/ plans/ .claude/   docs/ + 1_명세서/   🟠 운영
⑦   메신저 어댑터   카카오톡 단일 포맷   Win/Mac/슬랙 3종 어댑터   🟡 중요
① ② ③은 동시에 구현이 불가능한 수준의 충돌입니다.

①번은 PyInstaller --onefile 패키징과 직결됩니다. KoNLPy는 JDK를 런타임 의존성으로 요구하는데, JDK를 .exe 안에 번들하는 건 사실상 불가능합니다. 반면 kiwipiepy는 C 확장이라 번들 가능합니다. 즉 KoNLPy를 선택하면 단일 파일 실행(A-1 제약)을 포기해야 합니다.

②번은 알고리즘 선택의 문제가 아니라 구현 방향 자체가 다릅니다. Capping+log는 규칙 기반(하드코딩된 임계치)이고, IsolationForest는 데이터 기반(팀원 분포를 학습)입니다. 팀원 2~10인(A-5)이라는 소규모에서 IsolationForest가 의미 있는 결과를 내려면 contamination 파라미터 튜닝이 필요하고, 이게 FR-4.2에 명세돼 있지 않습니다.

③번은 사용자 경험 철학이 다릅니다. "재시작해도 이전 결과를 보여줘야 한다(v2.1 NFR-2.4)"와 "종료 시 모든 민감데이터를 파기해야 한다(새 명세서 N2.3)"는 양립 불가능합니다.

ADR-0004: NLP 라이브러리 선택

옵션 A: kiwipiepy 유지 → 단일 .exe 배포 가능, JDK 불필요
옵션 B: KoNLPy 채택 → 형태소 분석 품질 우수, JDK 설치 필수, 단일 파일 배포 불가
결정 기준: 단일 실행파일(C-6) 제약을 지킬 것인가 말 것인가
ADR-0005: 이상치 처리 방식

옵션 A: Capping + log scale → 단순, 예측 가능, 팀원 수 무관
옵션 B: IsolationForest + Z-Score → 데이터 기반, 팀원 2~10인 소규모에서 불안정 가능성 있음
옵션 C: 병행 — Capping은 데이터 전처리, IsolationForest는 경보 판단에만 사용
결정 기준: 구현 복잡도 vs 탐지 정확도 트레이드오프
ADR-0006: 캐시/데이터 생명주기

옵션 A: 원자적 파일 캐시 유지 → 재시작 후 결과 복원 가능
옵션 B: 종료 시 전체 휘발 → 프라이버시 강화, 매번 재분석 필요
결정 기준: 조장이 반복적으로 결과를 볼 필요가 있는가

## 4. 기능 요구사항 (FR)

### FR-1: OOXML 문서 분석 파이프라인 *(조원희)*

#### FR-1.1 — .pptx / .docx 글자 수 추출
- 구현
  - `python-pptx`, `python-docx`로 XML 파싱
  - .pptx: `prs.slides`의 모든 `shape.text_frame.text` 순회
  - .docx: `document.paragraphs`의 `paragraph.text` 순회
  - 공백·개행 제외 순수 글자 수: `len(text.replace(' ', '').replace('\n', ''))`
  - 도형(shape) 개수 별도 카운트
- 수용 기준
  - 200자 .docx → `200 ± 0`자
  - 슬라이드 3장 × 텍스트박스 2개(각 50자) → 글자 `300`, 도형 `6`
  - 빈 파일 → 글자 `0`, 도형 `0`, 예외 없음

#### FR-1.2 — 팀원 식별자 기반 집계
- 구현
  - 결과 스키마: `Dict[str, int]` = `{"팀원이름": 총_글자수}`
  - 작성자: .docx → `core_properties.author`, .pptx → `core_properties.last_modified_by`
  - 식별자가 `""` 또는 `None`이면 `"Unknown"` 키로 분류
- 수용 기준
  - Alice 100자 + Bob 150자 → `{"Alice":100, "Bob":150}`
  - 메타데이터 없음 → `"Unknown"` 키 존재, 예외 없음

### FR-2: Git 분석 파이프라인 *(조원희)*

#### FR-2.1 — git log 커밋 데이터 추출
- 구현
  - 호출: `subprocess.run(["git","log","--numstat","--format=%H|%ae|%ai"], cwd=repo_path, capture_output=True, text=True, timeout=30)`
  - 스키마: `{"author_email": {"commits": int, "additions": int, "deletions": int}}`
  - `subprocess.CalledProcessError`·`FileNotFoundError` → 빈 딕셔너리 `{}` 반환, 예외 상위 전파 금지
  - 타임아웃 **30초 고정**
- 수용 기준
  - 커밋 3개(`alice@test.com`, +10/-5) → `{"alice@test.com": {"commits":3, "additions":10, "deletions":5}}`
  - 잘못된 경로(예: `C:\not_a_repo`) → `{}`, 예외 없음
  - 10,000 커밋 저장소 → 30초 이내 완료

#### FR-2.2 — Git 헬스체크 + 안내 팝업
- 구현
  - 메인 윈도우 표시 **전** `subprocess.run(["git","--version"], timeout=5)` 1회 호출
  - `FileNotFoundError` 또는 반환 코드 `!= 0` → 모달 팝업 표시
    - 경고 문구: `"Git이 설치되어 있지 않거나 PATH에 등록되지 않았습니다."`
    - 다운로드 링크: `https://git-scm.com/download/win` (기본 브라우저로 열기)
    - PATH 설정 방법 안내 1줄 이상
    - `[확인]` 버튼 → 팝업 닫고 Git 분석 기능만 비활성화한 채 앱 계속 실행
- 수용 기준
  - Git 미설치 환경 → 메인 윈도우 표시 전 팝업 표시
  - 확인 클릭 후 앱 정상 실행
  - Git 정상 설치 → 팝업 미표시

### FR-3: 메신저 대화 로그 분석 *(조원희)*

#### FR-3.1 — 카카오톡 / 슬랙 파싱
- 구현
  - 카톡 `.txt` 정규식
    - 날짜: `r"^\d{4}년 \d{1,2}월 \d{1,2}일"`
    - 발화: `r"^(.+?)\s:\s(.+)$"`
  - 슬랙 `.csv`: `csv.DictReader`, 필수 컬럼 `user`, `text`, `timestamp`
  - 반환: `[{"author": str, "timestamp": str, "message": str}, ...]`
  - 인코딩: UTF-8 우선 → 실패 시 CP949(EUC-KR) 재시도 (NFR-3.1)
- 수용 기준
  - 카톡 형식 정확 파싱
  - 슬랙 CSV 3행 → 리스트 길이 `3`
  - CP949 한글 깨짐 없음

#### FR-3.2 — 오염 줄 Skip 방어 파싱
- 구현
  - 줄 단위 순회 + `try-except Exception` 블록
  - 예외 발생 시 `skip_count += 1` + `continue`
  - 반환: `{"records": List[Dict], "skipped_lines": int}`
- 수용 기준
  - 정상 10 + 오염 3 → `records` 길이 10, `skipped_lines` 3
  - 전체 100 오염 → `records=[]`, `skipped_lines=100`, 예외 없음
  - 두 경우 모두 정상 return, 프로세스 종료 없음

### FR-4: 최종 기여도 산출 알고리즘 *(김휘중)*

#### FR-4.1 — Min-Max 정규화
- 구현
  - 공식: `normalized = (x - min) / (max - min)`
  - `max == min` (전원 동일) → 전원 `0.5` 반환
  - 대상 지표 3종: Git 추가 라인, OOXML 글자 수, 메신저 발화 횟수
  - 결과는 `round(_, 4)` 적용
- 수용 기준
  - `[0, 50, 100]` → `[0.0, 0.5, 1.0]`
  - `[75, 75, 75]` → `[0.5, 0.5, 0.5]`, `ZeroDivisionError` 없음
  - 모든 결과 `0.0 ≤ v ≤ 1.0`

#### FR-4.2 — 이상치 Capping + 로그 스케일링
- 구현
  - 단일 커밋 추가 라인 > 1,000줄 → `1,000`으로 강제 Capping
  - Capping 이후 합산 → `math.log1p(x)` 적용 → 정규화
  - Capping 적용 커밋 1개 이상 시 반환에 `"capping_applied": True` 포함
- 수용 기준
  - 단일 커밋 5,000 입력 → 내부 집계값 1,000
  - 999 입력 → 999 그대로 사용 (Capping 없음)
  - Capping 발생 시 반환 dict에 `"capping_applied": True`

#### FR-4.3 — 메신저 결측 시 동적 가중치 재조정
- 구현
  - 기본 가중치 예시: `W_git=0.4, W_doc=0.4, W_msg=0.2` (FR-4.4 균형 프리셋)
  - 메신저 데이터 부재 → `W_msg = 0.0`
  - 나머지 재조정: `W_i_new = W_i / (1.0 - W_msg)`
    - 재조정 결과: `W_git_new=0.5, W_doc_new=0.5`
  - 가중치 합 `round(sum, 4) == 1.0` assert로 내부 검증
  - 최종 점수 = `Σ(normalized_i × W_i_new)`, 4자리 반올림
- 수용 기준
  - 메신저 입력 `None`일 때 예외 없이 점수 dict 반환
  - 내부 가중치 합 `1.0 ± 0.0001`
  - 메신저 정상 vs 부재 → **다른** 점수 반환

#### FR-4.4 — 직무별 가중치 프리셋 + 슬라이더
- 프리셋 (UI 버튼 3개)

  | 프리셋 | W_git | W_doc | W_msg |
  |---|---|---|---|
  | 개발 중심 | 0.60 | 0.25 | 0.15 |
  | 기획 중심 | 0.20 | 0.60 | 0.20 |
  | 균형 설정 | 0.40 | 0.40 | 0.20 |
- 슬라이더: 각 지표 `0.00 ~ 1.00`, 스텝 `0.05`
- 합 ≠ 1.0 → `[분석 시작]` 비활성화 + `"가중치 합계가 1.00이어야 합니다. 현재: X.XX"`
- 수용 기준
  - `[개발 중심]` 클릭 → 슬라이더 `0.60 / 0.25 / 0.15`
  - 합 1.5 → 버튼 비활성화 + 경고 표시
  - 합 1.0 → 버튼 활성화

### FR-5: 시각화 및 대시보드 *(이대한)*

## 5. 비기능 요구사항 (NFR)

### NFR-1: UI 비동기 처리 *(이대한)*
- 구현
  - 파싱·분석 전체를 메인 스레드와 분리된 **Worker Thread** (`QThread+QRunnable`, `threading.Thread` 등)에서 실행
  - 시작 시 진행률 표시줄(Progress Bar) 표시
  - 완료/오류 시 Signal/Callback으로 메인 스레드에 통보
- 수용 기준

  - 시작 1초 이내 진행률 표시
  - 완료·오류 시 진행률 사라짐 + 결과/메시지 표시

### NFR-1.2: 분석 중복 실행 방지 (Race Condition Guard)
- 구현
  - 전역 플래그 `is_analyzing: bool = False`
  - `[분석 시작]` 클릭 시 `True`면 즉시 `return`
  - Worker 시작 직전: `is_analyzing=True` + 버튼 `setEnabled(False)`
  - Worker 종료 시 (성공·오류·취소 무관): `is_analyzing=False` + 버튼 `setEnabled(True)`
  - 상태 복원은 메인 스레드 Signal/Callback에서만 (Worker 내부 직접 UI 수정 금지)
- 수용 기준
  - 시작 후 **0.5초** 이내 버튼 비활성화
  - 5회 연속 클릭 → Worker 1개만 실행 (`threading.active_count()` 검증)
  - 재분석 결과가 첫 실행과 동일 (중복 누적 없음)
  - `RuntimeError`로 비정상 종료 시에도 버튼 재활성화

### NFR-2: 데이터 보안 및 프라이버시

#### NFR-2.1 — 읽기 전용 파일 접근
- 분석 대상은 `'r'` / `'rb'` 모드만 사용. `'w'/'a'/'x'` 절대 금지
- 리포트 저장은 사용자가 지정한 새 경로에만 쓰기
- 수용 기준
  - 입력 원본 파일 `os.path.getmtime`이 분석 전후 동일
  - 코드 리뷰 시 분석 대상에 `open(path,'w')` 또는 `shutil.move/delete` 발견 → Fail

#### NFR-2.2 — 네트워크 통신 완전 차단
- `requests`, `urllib`, `httpx`, `socket`, `http.client` 임포트 금지
- 링크 열기는 `webbrowser.open()` (OS 위임)만, 앱 내부 HTTP 요청 생성 금지
- 수용 기준
  - 방화벽·Wireshark로 외부 IP 송신 패킷 **0건**
  - 코드에 네트워크 라이브러리 import 발견 → Fail

#### NFR-2.3 — 익명화 캐시 (원자적 쓰기)
- 직렬화: `json.dumps()`**만** 사용. `pickle` 모듈 import 전면 금지
- 캐시 저장 **허용** 데이터: 팀원 식별자(이름·이메일), 정규화 점수, 분석 타임스탬프, 가중치 설정값
- 캐시 저장 **금지** 데이터: 메시지 본문(`str`), 소스코드 파일 내용
- 원자적 쓰기 3단계
  1. 동일 디렉터리 임시 파일 `.qce_cache.tmp`에 JSON 완전 쓰기
  2. `tmp_file.flush()` + `os.fsync(tmp_file.fileno())`로 디스크 기록 보장
  3. `os.replace('.qce_cache.tmp', '.qce_cache')` 단일 원자 교체
- 손상 캐시 복구
  1. 로드 중 `json.JSONDecodeError` 또는 `KeyError` → `.qce_cache` 즉시 `os.remove`
  2. 모달: `"캐시 파일이 손상되어 삭제되었습니다. 재분석이 필요합니다."`
  3. 빈 초기 상태로 앱 정상 시작
- 수용 기준
  - 캐시 `json.loads()` 시 점수 dict 정상 복원
  - 캐시 내에 메신저 원본 메시지 문자열 존재 → Fail
  - 코드에 `import pickle` 또는 `from pickle` 발견 → Fail
  - 쓰기 중 `taskkill /F` 후 재시작 → 다이얼로그 표시 + 초기 화면
  - 정상 쓰기 후 `.qce_cache.tmp` 잔존 → Fail

#### NFR-2.4 — 원본 파일 없이 캐시만으로 로드
- `.qce_cache` 존재 시 원본 파일 검사 없이 즉시 역직렬화 → 대시보드 표시
- 상태바: `"캐시 파일에서 이전 분석 결과를 불러왔습니다. (분석 일시: YYYY-MM-DD HH:MM)"`
- 수용 기준
  - 원본 `.txt`, `.docx`, Git 저장소 폴더 전체 삭제 후 재시작 → 차트·점수 정상 표시
  - 상태바 캐시 로드 메시지 표시

### NFR-3: 안정성·테스트 가능성

#### NFR-3.1 — 인코딩 자동 감지
- 시도 순서
  1. `open(path,'r',encoding='utf-8')`
  2. `UnicodeDecodeError` 시 `encoding='cp949'` 재시도
  3. 둘 다 실패 → 파일 skip + `{"error":"encoding_failed","path":path}` 반환

- 수용 기준

  - CP949 → UTF-8 실패 후 CP949로 성공, 한글 정상
  - Shift-JIS 등 → 종료 없이 `error` 키 dict 반환

#### NFR-3.2 — 메신저 모듈 결합도 완화 (격리 테스트)

- 수용 기준

배경

교수님이 5천 줄 이상 프로그램을 요구한 건 사람 손으로만 짜는 게 불가능한 규모를 줘서 AI를 써서 완성까지 가는 경험을 해보게 하려는 의도
"기능적으로 작동하면 된다"고 하신 것도 같은 맥락 — 완성도보다 완성까지 가는 과정을 평가하는 것


테스트 문서 검토 결과

test-plan.md, test-cases.md 두 문서를 검토했고, 학부 수준을 크게 넘는 품질로 잘 작성됨
경계값(예: Capping 999/1000/1001줄), 비정상 입력(깨진 파일, 빈 파일, 한글 인코딩) 케이스가 모두 포함됨
모든 요구사항이 어떤 테스트로 검증되는지 100% 매핑되어 있음
단, 문서는 아직 한 줄도 실행되지 않은 상태였음 → 그래서 실제 실행 확인이 필요했음


테스트 실행 시도 및 결과

이 채팅에서 직접 실행을 시도했지만 네트워크가 막힌 샌드박스 환경이라 패키지 설치 자체가 불가능했음
Claude Code(로컬 설치형)를 사용하면 해결된다는 것을 확인 → 실제로 설치 후 실행


역할 분담 방향

기존의 Backend / Business Logic / Frontend 역할 분담은 순서 충돌 문제가 있음

예: Frontend 담당자는 Backend가 완성될 때까지 아무것도 못 함


대신 STAGE 순서 기반으로 역할을 나누는 것이 더 효율적

  김휘중 → STAGE 1, 4, 5  (수치 계산·분석 로직)
  조원희 → STAGE 2, 3     (파일 파싱)
  이대한 → STAGE 7        (UI·차트, STAGE 4 완료 후 시작)

단, 김휘중의 STAGE 1이 먼저 완료되어야 나머지 두 명이 시작 가능


현재 진행 상태

STAGE 0 (코드 규칙 자동 검사) + STAGE 1 (핵심 수치 계산) 34개 테스트 전부 통과(GREEN)
완성된 파일 목록:

types.py — 데이터 구조 정의
normalizer.py — 점수 정규화
capping_scaler.py — 이상 기여도 제한
weight_preset_manager.py — 가중치 프리셋
weight_rebalancer.py — 데이터 소스 결측 시 가중치 재계산




앞으로 남은 순서

STAGE 2~3: 파일 파서 구현 (문서, Git, 카카오톡) → 조원희
STAGE 4~5: 분석 통합 + 결과 저장·내보내기 → 김휘중
STAGE 6~7: 화면 연결 + 차트 3종 구현 → 이대한
STAGE 8: 전체 통합 테스트
빌드: PyInstaller로 .exe 파일 생성 → 제출

제일 중요한 결정입니다. 배경부터요.
차트를 그리려면 팀원별 점수 묶음이 필요해요. 그게 MemberScore라는 데이터 덩어리입니다 (이름, git 점수, 문서 점수, 종합 점수 같은 걸 담은 한 사람치 카드라고 보면 됩니다). 그런데 이 MemberScore는 원래 Model(계산하는 쪽)이 만들어내는 물건이에요.
문제는 이 프로젝트의 절대 규칙(C-4)이 **"View는 Model을 절대 import하면 안 된다"**는 거예요. 정적 분석으로 "View가 Model을 가져다 쓴 횟수 = 0"이어야 통과입니다. 그런데 차트(View)가 MemberScore를 쓰려고 from model import MemberScore 하는 순간 이 규칙을 어기게 됩니다.
그래서 "그럼 MemberScore를 누구 소유로 둘 거냐"를 정해야 합니다. 두 가지 길이 있어요.

제 제안: MemberScore를 Model도 View도 아닌 중립 지대(common/dto.py 같은 공용 폴더)에 둡니다. 그러면 Model도 거기서 꺼내 쓰고 View도 거기서 꺼내 쓰니까, "View가 Model을 import한" 게 아니게 됩니다. 규칙 위반이 아니에요. 양쪽이 공용 창고를 쓰는 셈.
더 빡센 대안: 차트한테 MemberScore 같은 깔끔한 객체를 아예 안 주고, Controller가 {"이름": "홍길동", "git": 0.5, ...} 같은 **그냥 딕셔너리(dict)**로 풀어서 넘깁니다. 차트는 점수 객체가 뭔지조차 모르고 그냥 숫자만 받아요. 더 안전하지만 코드가 좀 지저분해지고(score.git_score 대신 data["git"]), 점수 항목이 9개라 손이 더 갑니다.
