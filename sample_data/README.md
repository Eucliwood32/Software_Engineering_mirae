# QCE 예시 데이터셋 — 가상 팀 프로젝트 "도서관 대출 관리 시스템"

조장(사용자) 입장에서 QCE를 실제로 써 보거나 시연할 때 입력으로 사용하는 예시 데이터다.
ConOps 시나리오 A(학기말 종합 평가)를 그대로 재현하며, **문서·Git·메신저** 3종 입력을 모두 포함한다.

> 이 데이터는 모두 가상이며 실제 인물·저장소와 무관하다. `make_sample_data.py`로 언제든 재생성할 수 있다.

---

## 1. 가상 팀 구성 (4인)

| 팀원 | 역할 | Git 식별자(email) | 문서 작성자 | 카톡 표시명 | 의도한 시연 포인트 |
|---|---|---|---|---|---|
| **박서준** | 조장·백엔드 | `박서준` | 박서준 | 박서준 | 최상위 기여(기준선) |
| **최민호** | 문서·발표 | `최민호` | 최민호 | 최민호 | **EW-02 버스트 커밋** |
| **정유나** | 보조 | `정유나` | (문서 없음) | 정유나 | **Z-score 하위 이상치** |
| **이지은** | 프론트엔드 | `jelee@lib.dev` | `Jieun Lee` | 이지은 | **계정 병합**(식별자 분산) |

- 박서준·최민호·정유나는 Git email을 한글 이름으로 두어 문서·카톡 식별자와 **자동으로 합쳐진다**.
- **이지은만** 소스마다 식별자가 달라(`jelee@lib.dev` / `Jieun Lee` / `이지은`) 자동으로 묶이지 않는다.
  → 결과 화면에서 조장이 **수동으로 한 사람으로 병합**해야 진짜 기여도가 드러난다.

---

## 2. 파일 구성

```
sample_data/
├── docs/                         # 문서 8개 (docx 4 · pptx 3 · hwpx 1)
│   ├── 기획서.docx               # 박서준
│   ├── 요구사항명세서.docx        # Jieun Lee (= 이지은)
│   ├── 설계서.docx               # 최민호
│   ├── 회의록.docx               # 최민호
│   ├── 중간발표.pptx             # 최민호
│   ├── 최종발표.pptx             # Jieun Lee (= 이지은)
│   ├── 시연시나리오.pptx          # 박서준
│   └── 사용자매뉴얼.hwpx          # 최민호
├── chat/
│   └── kakao_chat.txt            # 9~12월 단톡방 대화 (카카오톡 export 형식)
├── repo/                         # Git 저장소 (커밋 33개, 9~12월)
└── make_sample_data.py           # 위 데이터를 만드는 생성 스크립트
```

---

## 3. 재생성

```bash
python sample_data/make_sample_data.py
```
실행 시 `docs/`·`chat/`·`repo/`를 비우고 새로 만든다. (`python-docx`, `python-pptx` 필요)

---

## 4. GUI 시연 절차 (전체 기능 — 권장)

```bash
python main.py
```

1. **입력 적재** — 제출 화면에 `docs/`의 문서 8개와 `chat/kakao_chat.txt`를 **드래그&드롭**하고,
   **[Git 저장소 선택…]**으로 `sample_data/repo`를 지정한다.
2. **가중치 설정** 후 **[분석 시작]**.
3. **1차 결과 화면** — 식별자 6개가 보인다. 이지은이 `jelee@lib.dev`·`Jieun Lee`·`이지은`
   **3개로 분리**되어 각각 약해 보이고, **최민호에 EW-02(버스트) 신호**가 표시된다.
4. **신원 매핑(계정 병합)** — 매핑 컨트롤에서 그 3개 식별자를 모두 **`이지은`**으로 지정한다
   (나머지 3인은 각자 본인 이름으로 지정; 미지정 행은 붉게 강조됨) → **확인**.
5. **재집계된 결과** — 4인으로 정리된다.
   - **이지은**이 git+문서+카톡 합산으로 **상위 기여자**로 올라온다(병합 효과).
   - **정유나**에 **Z-score 하위 이상치** 신호가 뜬다(모든 축 최저).
   - **최민호**의 **EW-02** 신호는 유지된다.
6. 막대·레이더·산점도 3종 차트와 신호 패널을 확인하고, **파일 ▸ 리포트 저장**으로 .md/.csv/.html/.json 내보내기.

### 병합 후 예상 결과 (균형 설정 0.4/0.4/0.2 기준)

| 팀원 | 종합 | Git | 문서 | 메신저 | 신호 |
|---|---|---|---|---|---|
| 최민호 | 0.87 | 0.87 | 1.00 | 0.63 | **EW-02** |
| 박서준 | 0.83 | 1.00 | 0.58 | 1.00 | — |
| 이지은 | 0.80 | 0.93 | 0.73 | 0.69 | — |
| 정유나 | 0.00 | 0.00 | 0.00 | 0.00 | **ZSCORE** |

> 정유나의 종합이 0.00인 것은 모든 지표에서 최저값이라 Min-Max 정규화상 0이 되기 때문이며,
> 결측이 아니라 "현저히 낮은 기여"를 의미한다(원시값: 커밋 30줄·메시지 3건).

## 5. CLI 사용 (참고)

```bash
python -m qce --git sample_data/repo \
  --doc sample_data/docs/기획서.docx --doc sample_data/docs/요구사항명세서.docx \
  --doc sample_data/docs/설계서.docx --doc sample_data/docs/회의록.docx \
  --doc sample_data/docs/중간발표.pptx --doc sample_data/docs/최종발표.pptx \
  --doc sample_data/docs/시연시나리오.pptx --doc sample_data/docs/사용자매뉴얼.hwpx \
  --msg sample_data/chat/kakao_chat.txt --weights 0.4,0.4,0.2
```

- **CLI에는 계정 병합 단계가 없다.** 따라서 이지은이 `jelee@lib.dev`/`Jieun Lee`/`이지은` 3개로
  분리되어 출력되는 것이 정상이며, **계정 병합과 깔끔한 Z-score는 GUI에서만** 확인된다.
- **최민호의 EW-02(버스트)** 신호는 CLI에서도 그대로 표시된다.
