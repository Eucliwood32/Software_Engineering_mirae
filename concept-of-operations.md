## 1. 문서 개요

### 1.1 목적
본 문서는 QCE(Quantitative Contribution Evaluator) 시스템이 **누구를 위해, 어떤 환경에서, 어떻게 운영되는지**를 이해관계자 관점에서 정의한다. SRS가 "무엇을 만들 것인가(What)"를 다룬다면, ConOps는 "어떻게 사용될 것인가(How it is used)"를 다룬다.

### 1.2 적용 범위
- 시스템: QCE 데스크톱 애플리케이션 (Python 3.10+ Standalone)
- 운영 주체: 팀 프로젝트 조장 (단일 사용자)
- 운영 시점: 학기 중 중간/기말 평가, 또는 임의의 정량 평가 시점
- 운영 환경: 조장 개인 PC, Air-Gapped 단독 실행

### 1.3 본 문서가 다루지 않는 것
- 구체적 함수 시그니처, 클래스 다이어그램 → `02-design-planning/` 참조
- 테스트 케이스 정의 → `03-verification-validation/` 참조
- 빌드/패키징 절차 → `06-implementation/build-and-package.md` 참조

### 1.4 참조 문서
- SRS v1.0 (본 저장소 루트 통합 명세서)
- `01-requirements/requirements-record.md`
- `05-decisions/ADR-0001-MVC.md`, `ADR-0002-PyQt6.md`, `ADR-0003-IsolationForest.md`

### 1.5 용어
SRS의 용어 정의를 우선 따르며, 본 문서에서 추가로 사용하는 용어는 다음과 같다.

- **운영 세션(Operational Session)**: 조장이 QCE를 실행한 시점부터 종료할 때까지의 단일 사용 단위. NFR-2.3에 따라 세션 종료 시 민감 데이터는 휘발 파기된다.
- **평가 사이클(Evaluation Cycle)**: 입력 수집 → 매핑 → 분석 → 리포트 검토 → 판단까지의 1회 평가 흐름.
- **어뷰징(Abusing)**: 자동 생성 코드 일괄 커밋, 비정상적 대량 푸시 등 정량 지표를 왜곡할 의도/결과의 행위.

## 2. 현재 상황 (As-Is)

### 2.1 현재의 평가 방식
QCE 도입 이전, 조장이 팀 기여도를 평가하는 방식은 일반적으로 다음 중 하나에 해당한다.

1. **주관 평가**: 조장이 "체감상" 누가 얼마나 기여했는지 판단. 객관적 근거 부재.
2. **수기 집계**: GitHub 커밋 수를 수동으로 세고, 카톡 대화량을 눈으로 훑고, 발표자료 작성자를 기억에 의존해 정리.
3. **단순 자가 보고**: 팀원이 직접 기여도를 % 형태로 보고하고 조장이 조정.

### 2.2 As-Is의 한계
현재 방식의 구조적 문제는 다음과 같다.

- **L1. 통합 불가**: 코드 기여, 문서 작성, 메신저 소통이 각각 다른 도구·플랫폼에 흩어져 있어 단일 지표로 통합 비교가 불가능하다.
- **L2. 어뷰징 무방비**: AI 생성 코드를 통째로 커밋하거나, 마감 직전 대량 푸시로 활동량을 부풀리는 행위를 자동으로 탐지할 수단이 없다.
- **L3. 신원 분산**: 한 팀원이 GitHub `dh-lee`, 카톡 `이대한`, 슬랙 `daehan.lee` 등 플랫폼마다 다른 식별자를 가져 통합 추적이 어렵다.
- **L4. 메신저 잡음**: "ㅇㅇ", "ㅋㅋ", 이모티콘 등이 메시지 수에 포함되어, 실제 협업 소통량과 형식적 응답을 구분하지 못한다.
- **L5. 시각화 부재**: 데이터를 모아도 비교 가능한 형태로 시각화하는 도구가 없어 의사결정 비용이 크다.
- **L6. 시간 비용**: 위 작업을 수기로 처리하면 1회 평가에 수 시간이 소요되며, 결과의 일관성도 보장되지 않는다.

### 2.3 As-Is의 이해관계자 통증점
- **조장**: 평가 시간 부담, 판단 근거의 객관성 부족, 팀원과의 형평성 시비 위험.
- **팀원**: 기여도가 정성 평가에 좌우되어 노력이 가시화되지 않는다는 불만 가능성.
- **교수/조교**: 조장의 보고를 검증할 객관적 자료가 부족.

## 3. 변경의 필요성과 본질 (Justification)

### 3.1 왜 지금 변경하는가
- 팀 프로젝트 규모가 커지고 협업 채널이 다변화(Git, 메신저, 문서)되면서, 단일 채널 관찰로는 기여도를 측정할 수 없다.
- 생성형 AI 도구 보급으로 코드/문서 어뷰징 가능성이 구조적으로 증가했다.
- 동시에, 외부 LLM API에 팀 데이터를 업로드하는 것은 프라이버시·보안 측면에서 부적절하다.

### 3.2 변경의 본질
QCE는 **"수기 → 자동 통합 + 어뷰징 방어 + 시각화"**로의 전환이며, 다음 4대 목표(SRS O-1~O-4)로 환원된다.

- O-1 통합분석: `.pptx/.docx` + 메신저 로그 + Git을 단일 파이프라인으로 결합
- O-2 어뷰징 방어: 자동 생성 코드·비정상 커밋의 통계적 탐지·경고
- O-3 시각화: 작업 지분율·소통량×작업 밀도 등 다각도 리포트
- O-4 완전 로컬: 외부 AI/클라우드 없이 조장 PC에서 단독 작동

### 3.3 명시적 비목표 (재확인)
- 실시간 협업 모니터링 아님 (배치성 정량 평가 도구)
- LLM 기반 의미 분석 아님 (규칙 기반 NLP)
- 팀원 PC에 에이전트를 설치하지 않음
- 외부 서버로 데이터를 전송하지 않음

## 4. 제안 시스템 (To-Be) 개요

### 4.1 시스템 한 줄 요약
> 조장의 PC에서 단독 실행되며, 문서·코드·메신저 3종 입력을 한 번에 받아 팀원별 정량 기여 지표와 어뷰징 경보를 시각화 리포트로 제공하는 데스크톱 도구.

### 4.2 운영 모드
QCE는 단일 운영 모드를 가진다(다중 모드 없음).

- **분석 세션 모드**: 조장이 입력 데이터를 모아 적재 → 매핑 → 분석 → 리포트 조회를 일괄 수행. 세션 종료 시 NFR-2.3에 따라 민감 데이터는 메모리·임시 파일에서 모두 파기된다.

### 4.3 To-Be의 가치 명제
| As-Is 한계 | To-Be 해결 수단 | 관련 FR |
| --- | --- | --- |
| L1 통합 불가 | 3종 입력 단일 파이프라인 + 정규화 특성 매트릭스 | FR-1, FR-2, FR-3, FR-4.1 |
| L2 어뷰징 무방비 | EW-01(≥1000줄 단일 커밋), EW-02(빈도×3 폭주) | FR-2.2 |
| L3 신원 분산 | Global Alias Mapping UI | FR-1.3 |
| L4 메신저 잡음 | 불용어 사전 + 형태소 분석 + 유효 글자수 산출 | FR-3.2 |
| L5 시각화 부재 | 파이/산점/타임라인/네트워크/레이더 시각화 | FR-5 |
| L6 시간 비용 | 전체 파이프라인 ≤300초(NFR-3.3) | NFR-3 |

## 5. 운영 환경

### 5.1 물리 환경
- 조장의 일반 데스크톱 또는 노트북 1대 (특수 하드웨어 요구 없음)
- 인터넷 연결은 빌드/설치 단계 이후에는 불필요 (Air-Gapped 동작 가능, NFR-1)

### 5.2 시스템 환경
- OS: Windows 10/11 또는 macOS 12+
- 런타임: Python 3.10+ 또는 PyInstaller/Nuitka 패키지 실행
- 종속: KoNLPy 사용을 위해 Java Runtime(JDK 11+) 사전 설치 (D-2)
- Git CLI: 로컬 저장소 분석 시 필요 (D-4) — `git log` 텍스트 파일만 입력하는 경우 미필요

### 5.3 자원 한계
- RAM ≤ 2GB 사용 (NFR-3.4)
- 처리 시간 상한: Git 500커밋 ≤60s, 카톡 10MB ≤120s, 전체 파이프라인 ≤300s (NFR-3.1~3.3)

### 5.4 데이터 환경
- 입력 데이터는 조장이 사전에 수집하여 로컬 디스크에 둔다(외부 자동 fetch 없음).
- 입력 형식: `.pptx/.docx`, `.txt/.csv`(메신저 export), 로컬 Git 저장소 또는 `git log` 텍스트.
- 산출 데이터: 메모리 내 DataFrame과 UI 화면에 한해 존재. 세션 종료 시 휘발.

## 6. 이해관계자 및 사용자 특성

### 6.1 1차 사용자 — 조장
- **인원**: 1명 (단일 사용자 전용, RBAC 미지원)
- **기술 친숙도**: 컴퓨터공학/정보 계열 학부생 수준 가정. CLI는 가능하나 GUI 친화 환경을 선호.
- **사용 빈도**: 학기당 2~4회 (중간·기말 평가 시점, 필요 시 중간 점검)
- **주요 관심사**: 평가의 객관성·재현성, 어뷰징 식별, 결과 시각화의 명확성, 결과 해석의 자율성(시스템이 결론을 대신 내리지 않을 것).

### 6.2 2차 이해관계자
| 이해관계자 | 관심 | QCE와의 관계 |
| --- | --- | --- |
| 팀원 (평가 대상, 2~10인) | 본인 기여가 공정하게 측정되는가 | 직접 사용은 하지 않으나, 입력 데이터의 생성자 |
| 교수/조교 | 조장 보고의 신뢰성 | 조장이 산출한 리포트를 수신할 수 있음 |
| 개발팀(본 프로젝트 3인) | 유지보수성, RTM 추적성 | NFR-4 및 RTM 정책 책임 |

### 6.3 사용자 특성 가정
- A-5에 따라 팀 규모는 2~10인 범위로 한정한다.
- 조장은 입력 데이터를 본인이 수집·배치할 책임이 있으며, QCE는 데이터 자동 수집을 수행하지 않는다.

### 6.4 사용자가 가지지 않는 권한
- 외부 서버 전송 기능 없음 (C-1, C-2)
- 팀원 PC에 에이전트를 설치하는 기능 없음 (C-3)
- 결과를 클라우드에 백업하는 기능 없음

## 7. 운영 시나리오

본 절은 QCE의 실제 사용 모습을 시간순 서사로 서술한다. 각 시나리오는 SRS의 유스케이스(UC-1~UC-9)와 매핑된다.

### 7.1 시나리오 A — 학기말 종합 평가 (메인 시나리오)

**상황.** 15주차 기말, 조장은 팀 4인의 학기 전체 기여도를 정량 평가해 교수에게 제출해야 한다. 조 구성은 다음과 같다.
- 본인(조장)
- B 팀원 (백엔드 담당)
- C 팀원 (프론트엔드 담당)
- D 팀원 (문서·발표 담당)

**선행 조건.**
- 팀 GitHub 저장소가 조장 PC에 로컬 클론되어 있다.
- 카카오톡 단톡방 대화가 학기 시작일부터 종강일까지 `.txt`로 export되어 있다.
- 발표 자료(`.pptx`) 3개, 기획서/보고서(`.docx`) 5개가 한 폴더에 모여 있다.

**흐름.**

1. **실행 (UC-1 진입).** 조장이 패키징된 `QCE.exe`를 더블클릭한다. 메인 윈도우가 뜨고, 좌측에 팀원 목록 패널, 중앙에 Drag&Drop 영역, 우측에 경보 영역이 표시된다(FR-1.1).
2. **문서 적재.** 조장이 8개 OOXML 파일을 한 번에 Drag&Drop으로 던진다. UI 파일 목록에 "8/8 적재 완료"가 표시되고, 내부적으로 `DocumentParser`가 `core.xml`/`app.xml`을 파싱해 각 파일의 `dc:creator`, `cp:lastModifiedBy`, byte 수정량을 추출한다(FR-1.2).
3. **Git 적재.** 조장이 메뉴에서 "Git 저장소 선택"을 눌러 로컬 클론 경로를 지정한다. `GitAnalyzer`가 `git log --numstat --format`을 실행해 커밋별 작성자·날짜·추가/삭제 행을 수집한다(FR-2.1). 500커밋 가정 시 처리 시간은 60초 이내(NFR-3.1).
4. **메신저 적재.** 조장이 카톡 export `.txt`를 Drag&Drop한다. `ChatAdapter`가 UTF-8 BOM과 "2026년 5월 27일 오전 1:00" 형식을 감지해 카카오톡 Windows 어댑터를 선택하고, `{timestamp, speaker, message}` 스키마로 정규화한다(FR-3.1).
5. **신원 매핑 (UC-2).** QCE가 발견한 식별자가 자동으로 매핑 다이얼로그에 나열된다. 예시 화면:
   - GitHub `dh-lee` → ?
   - GitHub `bk_cho` → ?
   - 카톡 `이대한` → ?
   - 카톡 `조원희` → ?
   - 문서 작성자 `Daehan Lee` → ?

   조장이 드롭다운에서 각 alias를 실제 팀원(B, C, D, 조장 본인)에 1:1로 매핑한다(FR-1.3). 미매핑 alias는 경고와 함께 "분석 제외" 옵션이 제시된다.
6. **분석 실행 (UC-5).** 조장이 "분석 시작" 버튼을 누른다. 내부적으로:
   - `ContributionEngine`이 팀원×지표 매트릭스를 구성하고 Min-Max/Z-Score 정규화 (FR-4.1)
   - `IsolationForest` 모델 훈련 → anomaly score (FR-4.2)
   - 지표별 Z-Score → Z<-1.5가 2개 이상인 팀원을 하위이상치로 판정 (FR-4.3)
   - `EarlyWarningEngine`이 EW-01, EW-02 룰 적용 (FR-2.2)
7. **리포트 조회 (UC-6).** 대시보드 탭이 열린다. 조장은 다음을 차례로 확인한다.
   - 작업 지분율 파이차트 (FR-5.1) — 조장 28%, B 31%, C 29%, D 12%
   - 소통량(X) × 문서 작업 밀도(Y) 산점도 + 평균선 (FR-5.2)
   - 타임라인 추이, 레이더차트 등 고급 시각화 (FR-5.3)
8. **경보 확인 (UC-7).** 우측 경보 영역에 다음이 표시된다.
   - ⚠️ EW-01: "B 팀원 3월 22일 단일 커밋 +1,847줄 — 대량 코드 변경 감지(자동 생성 의심)"
   - 산점도에서 D 팀원의 점이 붉은색 + ⚠️ 오버레이로 강조됨(FR-5.4) — Z-Score 하위 이상치 판정.
9. **판단 및 종료.** 조장은 경보를 검토하고, 필요 시 시나리오 B로 진행한다. 작업이 끝나면 QCE를 종료하고, 종료 직후 모든 메모리·임시 데이터가 휘발 파기된다(NFR-2.3).

**예상 소요 시간.** 입력 적재 ~3분, 매핑 ~2분, 분석 ~5분, 리포트 검토 ~10분. 총 약 20분 이내.

**대조 (As-Is).** 동일 작업을 수기로 했을 경우 1.5~3시간 예상. 어뷰징 탐지 불가.

### 7.2 시나리오 B — 어뷰징 의심 케이스 대응

**상황.** 시나리오 A의 8단계에서 B 팀원의 단일 커밋 1,847줄 경보를 받았다. 조장은 이를 검증해야 한다.

**흐름.**

1. **경보 항목 클릭.** 조장이 경보 카드를 클릭하면 해당 커밋의 상세 정보(커밋 해시, 작성일, 변경 파일 목록)가 표시된다.
2. **소스 확인.** 조장이 로컬 클론에서 해당 커밋을 직접 열어 확인한다. 결과는 다음 중 하나:
   - **케이스 B-1**: 정상이었음 — 프로젝트 초기 보일러플레이트(예: 생성된 라이브러리 일괄 추가). 자동 생성이긴 하나 어뷰징 의도는 없음.
   - **케이스 B-2**: 어뷰징 의심 — 학기 중반, 본인 작성이라 보기 어려운 대량 코드.
3. **케이스 B-1 처리 (UC-9).** 조장이 경보 카드의 "예외 처리" 버튼을 누른다. `EarlyWarningEngine`이 해당 커밋을 정상 처리로 표시하고, `ContributionEngine`이 점수를 재계산한다(FR-2.2 예외 처리). 산점도와 파이차트가 갱신된다.
4. **케이스 B-2 처리.** 조장은 경보를 유지한 채로 리포트를 export하고, 별도로 팀원과의 1:1 면담을 통해 사실 관계를 확인한다. 이때 QCE는 **판단을 대신하지 않는다** — 시스템은 통계적 이상 신호만 제시할 뿐, 어뷰징 여부의 최종 결론은 조장의 책임이다.

**중요 원칙.**
- 경보는 **혐의 제기가 아니라 검증 권유**다.
- 정상 처리된 예외는 다음 분석 사이클에도 유지될 수 있도록, 가능하면 예외 사유를 함께 기록한다(향후 확장 영역).

---

### 7.3 시나리오 C — 신원 매핑 갱신

**상황.** 학기 중반, 한 팀원이 GitHub 계정을 새로 만들어 커밋하기 시작했다. 기존 alias 매핑이 더 이상 유효하지 않다.

**흐름.**

1. **새 분석 세션 시작.** 조장이 QCE를 실행한다. 모든 세션 데이터는 휘발되었으므로(NFR-2.3) 매핑은 처음부터 다시 구성해야 한다.
2. **입력 적재.** 시나리오 A의 1~4단계와 동일하게 문서·Git·메신저를 적재한다.
3. **자동 식별자 추출.** `AliasMapper`가 모든 소스에서 사용된 식별자를 추출한다. 이번에는 GitHub 측에 새 계정 `bk-cho-2` 가 추가로 등장한다.
4. **매핑 다이얼로그 (UC-2).** 조장은 `bk-cho-2`도 기존 B 팀원에 매핑한다. 동일 팀원에 복수 alias가 매핑되는 것은 정상이며, AliasMapper는 이를 N:1 관계로 허용한다.
5. **저장 및 분석 진행.** 매핑 결과는 현재 세션에서만 유효하다 — NFR-2.4에 따라 OAuth Token 등은 디스크에 영구 저장하지 않으나, **alias 매핑 자체는 사용자의 작업 편의를 위해 별도 정책으로 관리할 수 있다**(설계 시 결정, ADR 후보).
6. **검증.** 분석 후 파이차트에서 B 팀원의 지분이 단일 인격으로 합산되어 표시되는지 확인한다.

**시나리오 C의 핵심.** 다플랫폼 사용자명 불일치 문제(As-Is L3)는 QCE의 핵심 가치 중 하나이며, 매핑 단계의 사용성은 운영 만족도를 좌우한다.

### 7.4 시나리오 D — 불용어 사전 커스터마이징

**상황.** 조장은 첫 분석 결과의 메신저 유효 글자수가 직관과 다르게 나왔다고 느낀다. 팀이 자주 쓰는 "굳굳", "ㄱㅅ"(감사), "ㅇㅋ"(오케이) 같은 표현이 유효 메시지로 잡혀 D 팀원의 소통량이 과대 평가된 것으로 보인다.

**흐름.**

1. **불용어 사전 진입 (UC-8).** 조장이 메뉴에서 "불용어 사전 편집"을 선택한다.
2. **카테고리별 편집.** 다음 카테고리가 노출된다(FR-3.2).
   - 단순긍정: 넵/네/넹/넴/응/웅/앙
   - 단순리액션: ㅇㅇ/ㅋㅋ/ㅎㅎ/ㄱㄱ/ㄴㄴ
   - 감탄·이해: 아하/아~/오오/헐/와
   - 미디어태그: (이모티콘)/(사진)/(동영상)
3. **추가.** 조장이 단순긍정 카테고리에 "굳굳", "ㄱㅅ", "ㅇㅋ"를 추가한다.
4. **재분석.** "분석 재실행" 버튼을 누른다. `NLPFilter`가 갱신된 사전으로 재필터링하고, 유효 글자수·유효 메시지수가 재계산된다. 산점도의 D 팀원 위치가 좌측(소통량 감소 방향)으로 이동한다.
5. **반복.** 결과가 직관과 부합할 때까지 사전 편집 → 재분석을 반복한다.

**시나리오 D의 핵심.** 규칙 기반 NLP의 한계는 사전 품질에 좌우되므로, 사전 편집 UX는 정확도의 핵심 통제 수단이다. LLM 기반 의미 분석을 도입하지 않은 트레이드오프(§9)와 직접 연결된다.

## 8. 운영 정책 및 제약

### 8.1 사용자가 반드시 인지해야 할 정책

- **P1. 데이터 휘발.** QCE 종료 시 모든 분석 결과·임시 파일이 메모리에서 즉시 파기된다(NFR-2.3). 리포트가 필요하면 종료 전 화면 캡처·export를 수행한다.
- **P2. 외부 전송 0.** QCE는 단 1바이트도 외부망으로 전송하지 않는다(NFR-2.2). 입력 데이터는 조장 PC를 벗어나지 않는다.
- **P3. AI 미사용.** QCE는 외부 LLM/AI API를 호출하지 않는다(C-2, NFR-1.1). 모든 판정은 결정론적 규칙·통계 모델 기반이다.
- **P4. 단일 사용자.** 다중 사용자 동시 사용은 지원하지 않는다. 한 PC에서 한 명의 조장만 사용한다.
- **P5. 판단의 책임.** QCE는 어뷰징·하위 이상치를 **탐지**하지만, **판정**하지 않는다. 최종 평가는 조장의 책임이다(시나리오 B 참조).

### 8.2 운영 제약 (SRS 제약 재확인)

| ID | 제약 | 운영 의미 |
| --- | --- | --- |
| C-1 | 로컬 CPU/RAM만 사용 | 클라우드 가속 옵션 없음 |
| C-2 | 외부 LLM 금지 | 의미 분석은 규칙 기반에 한정 |
| C-3 | 타 팀원 PC 설치 금지 | 자동 데이터 수집 불가 — 조장이 수동 적재 |
| C-4 | MVC 엄격 준수 | 운영 영향 없음(개발 제약) |
| C-5 | 명시 Python 스택 | 운영 영향 없음 |
| C-6 | 단일 실행 파일 | 설치/배포가 단순 — USB 전달 가능 |

### 8.3 가정의 운영 영향
- A-2: Git 로그를 텍스트 파일로만 받을 수도 있으므로, 팀원이 클론 권한이 없는 저장소에서도 사용 가능하다.
- A-4: 입력 OOXML이 손상된 경우 해당 파일은 skip되며 경고가 표시된다(FR-1.2 예외).
- A-5: 팀원 수 1인 또는 11인 이상은 명시적으로 미지원이다. 분석 결과의 통계적 유의성 문제와 직접 관계.

## 9. 영향 분석 (Summary of Impacts)

### 9.1 운영상 영향
- 조장은 학기 시작 시점에 데이터 수집 루틴을 정해야 한다(예: 매월 말 카톡 export 백업).
- 첫 사용 시 약 30~60분의 학습 시간이 필요할 것으로 예상한다(UI 익숙, 매핑·사전 편집 절차 숙지).
- 평가 1회당 소요 시간이 As-Is 대비 1/5~1/10로 단축된다(20분 대 1.5~3시간).

### 9.2 조직적 영향
- **팀원 측면**: 평가 방식이 정량화된다는 사실을 팀원에게 사전 고지하는 것이 권장된다. 이는 측정 자체가 행동에 영향을 미친다는 점(Goodhart's law)을 완화하기 위함이다.
- **공정성**: 모든 팀원에게 동일한 규칙·동일한 사전·동일한 모델이 적용되므로, As-Is 대비 형평성 시비가 줄어든다.
- **부작용 가능성**: 팀원이 지표 최적화(예: 의미 없는 커밋 늘리기) 행동을 할 가능성. 이는 어뷰징 경보(EW-01, EW-02)와 조장의 정성 검토로 완화된다.

### 9.3 윤리적 영향
- **오탐 가능성**: EW-01, EW-02는 통계적 임계치 기반이므로 정상 작업이 오탐될 수 있다. 시나리오 B의 예외 처리(UC-9)가 이를 완충한다.
- **프라이버시**: 메신저 대화가 분석 대상이라는 점에서, 조장은 팀원의 동의 또는 최소한의 사전 고지를 확보하는 것이 윤리적으로 권장된다. QCE는 외부 전송을 하지 않으나(P2), 조장 PC 내에서의 분석 자체가 정보 처리에 해당한다.
- **책임 소재**: P5에 따라 최종 판단은 조장에게 귀속된다. QCE는 의사결정 보조 도구이지 의사결정 주체가 아니다.

### 9.4 기술적 영향
- 조장은 JDK 설치, Git CLI 설치 등 종속을 사전에 갖춰야 한다(D-2, D-4).
- 학기 중 KoNLPy/형태소 분석기 버전을 변경하면 동일 입력에 대한 분석 결과가 달라질 수 있다 → 가능하면 학기 내 동일 버전을 유지한다.

## 10. 대안 및 트레이드오프

### 10.1 왜 LLM 의미 분석을 쓰지 않는가
- **대안**: OpenAI/Anthropic API로 메시지의 "내용성" 점수를 매기는 방법.
- **기각 이유**: (1) 외부망 전송이 NFR-2.2 위반 (2) 비결정론으로 재현성 손상 (3) 비용 발생 (4) 팀 프라이버시 침해 우려.
- **트레이드오프**: 의미 이해 능력 ↓ ↔ 결정론·로컬성·프라이버시 ↑. 사전 편집(시나리오 D)으로 정확도를 보완한다.

### 10.2 왜 단일 사용자인가
- **대안**: 팀원 각자가 자신의 데이터를 입력하는 멀티 사용자 모드.
- **기각 이유**: (1) C-3 위반 (2) RBAC·인증 시스템이 필요해져 범위가 폭증 (3) 학기 프로젝트 규모 초과.
- **트레이드오프**: 자동 수집 편의 ↓ ↔ 단순성·로컬성 ↑.

### 10.3 왜 Isolation Forest인가
- **대안**: Z-Score 단독, DBSCAN, One-Class SVM 등.
- **선택 이유**: 소표본(2~10인)에서도 동작, 비지도, 다차원 처리 용이, 임계치 자동/수동 양쪽 지원.
- **상세**: ADR-0003 참조.
- **트레이드오프**: 해석 가능성은 Z-Score가 우수하므로 두 방법을 **병행**한다(FR-4.2, FR-4.3 둘 다 채택).

### 10.4 왜 데스크톱 앱인가 (웹 아님)
- **대안**: Flask/FastAPI + 브라우저 UI.
- **기각 이유**: (1) 단일 사용자에 서버 모델은 과잉 (2) 브라우저 ↔ 서버 IPC가 Air-Gapped 시 오히려 복잡 (3) 단일 `.exe`/`.app` 배포(C-6)가 USB 전달에 적합.
- **트레이드오프**: 크로스 디바이스 접근성 ↓ ↔ 배포 단순성·로컬성 ↑.

### 10.5 미해결 / 향후 확장 후보
- 예외 처리 사유 기록의 영속화 (현재는 세션 단위 휘발).
- 학기 간 비교 (히스토리 영속화) — NFR-2.3과 충돌하므로 별도 설계 필요.
- Alias 매핑의 안전한 재사용 (해시 기반 저장 등).
